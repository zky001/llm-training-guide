# A3.3 Agentic RAG：检索也是工具

## 300 页的员工手册

公司给智能体派了个活：回答员工提问「我们的差旅报销上限是多少？」

答案就躺在 300 页的员工手册里。全塞进上下文？且不说塞不塞得下——上一节刚讲过，就算塞得下也不该塞：又贵、又慢、还会「上下文腐烂」。何况手册每月更新，总不能每次改版都重训模型（上篇第 5 章说过，知识进参数要靠训练，成本完全不成比例）。

经典解法就是 <Term id="rag">RAG</Term>（检索增强生成）：**先按问题从手册里检索出最相关的几段，塞进上下文，再让模型照着回答。** 模型不用背下手册，只要「开卷考试」。

## 经典 RAG：一条固定流水线

标准做法四步：

1. **切块**：把手册切成几百字一段的小块；
2. **向量化**：每块算一个向量存进向量库（又是上篇的[词向量](/docs/language-models/embedding)思想：意思相近、向量相近）;
3. **检索**：用户问题也变成向量，捞出最相似的 top-k 块；
4. **生成**：把这几块拼进上下文，模型据此回答。

注意它的结构：**检索固定发生一次、固定在回答之前**——用 A0 的话说，这是一条预定义的流水线（下一章会叫它「提示链」），不是智能体。

它便宜、快、大多数时候够用。但三个场景会翻车：

- **措辞对不上**：用户问「出差住宿标准」，手册写的是「差旅费用管理办法」——向量相似度救得了一部分，救不了全部；
- **答案分散**：「报销上限」在第 3 章、「特殊城市加成」在第 7 章附表——检索一次只能捞到一处，答案注定残缺；
- **来者不拒**：检回来的段落不相关，模型也会硬着头皮基于它回答——检索错误顺流而下变成回答错误。

## Agentic 升级：把检索交给循环

三个失灵场景有个共同病根：**检索只有一发子弹，打偏了没有补救。**

Agentic RAG 的升级思路简单到近乎显然——把检索从「固定的前置步骤」变成**模型自主使用的工具**（回扣 A1）：搜索工具进工具箱，循环开转（回扣 A0），于是模型可以：

- 觉得问题措辞不好搜，先**改写查询**再搜（「出差住宿标准」→「差旅费用 住宿 限额」）；
- 搜完发现信息不全，**再搜一轮**（先查到上限，再专门查「特殊城市」）;
- 对检回内容存疑，**交叉核对**另一个来源，或者干脆回答「手册里没有明确规定」。

没有任何新机制——**只是把检索工具交给了行动循环**，查什么、查几轮、信不信，从写死的代码变成了模型的运行时决策（A0.2 光谱上往右挪了一格）。

和长上下文的关系也顺便理清：上一节说窗口有限，本节的 RAG 是「**把对的东西放进窗口**」，而上篇 [8.2 的长上下文](/docs/frontier/long-context)是「把窗口做大」。两者互补而非替代——窗口再大，也不该把 300 页手册的原文全款塞入；检索再准，多轮对话的工作记忆也得靠窗口本身。

:::caution 升级不是免费的
Agentic RAG 更强，也更贵更慢：每多一轮检索决策就多一轮模型调用，简单问题上它和经典 RAG 效果一样、成本翻几倍——又是 A0.2 的老话：能简单就简单。更要紧的是，**检索质量仍是地基**：模型只能在检回来的内容里挑，库切得烂、索引建得差，再聪明的循环也是「垃圾进垃圾出」——上篇 4.1 数据清洗的教训，在这里换了个马甲重演。
:::

<DeepDive title="检索的技术栈速览">

**切块策略。** 按固定长度切最简单但会拦腰截断语义；按段落/标题切保留结构；相邻块留重叠窗口防止关键句恰好跨界。块大小是召回粒度与上下文占用的权衡（典型几百 token）。

**混合检索。** 向量检索抓语义（「住宿标准」≈「宿费限额」），关键词检索（BM25 类）抓精确匹配（产品型号、法条编号——向量检索恰恰容易糊掉这些）。生产系统普遍两路并跑再合并，各补各的盲区。

**重排序（rerank）。** 先用便宜的检索粗捞几十条候选，再用一个小模型对「问题-段落」逐对精细打分、取前几名。召回率和精度分两层各自优化——和推荐系统的召回-排序架构同构。

**引用与可验证性。** 让模型在答案里标注「出自手册第 3.2 节」，用户可以点开核对。这不只是体验优化：强制引用会显著抑制模型「脱稿发挥」——答案必须能落到某段原文上，幻觉空间就小了（上篇 5.6 幻觉缓解手段的一种落地形态）。

**评测要分层。** 检索命中率（该找到的段落找到了吗）和端到端答对率（最终答案对吗）必须分开测——否则你不知道错在检索还是生成，改进无从下手。这是 A7 章「评测驱动开发」的一次预演。

</DeepDive>

## 小结

:::tip 本节要点
- RAG = 开卷考试：知识放外部库，按题检索、塞进上下文再回答——知识更新不用动模型。
- 经典 RAG 是固定流水线，措辞错位、答案分散、来者不拒三处会翻车。
- Agentic RAG 把检索变成循环里的工具：改写重搜、多轮补搜、交叉核对——更强但更贵。
- 检索质量是地基：库烂则全烂，聪明的循环救不了糟糕的索引。
:::

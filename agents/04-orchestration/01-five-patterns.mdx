# A4.1 五种积木：生产系统的乐高说明书

## 一百个系统，五种形状

如果你有机会观察一百个真正跑在生产环境里的 LLM 系统——客服机器人、合同审阅、代码评审、报告生成——你会发现一件有趣的事：抛开业务细节，它们的**结构**翻来覆去就那么几种形状。

2024 年底，Anthropic 的《Building Effective Agents》把这层窗户纸捅破了：五种可组合的积木，覆盖生产系统的绝大多数形态。从此「这块用路由、那块套个评估器」成了行业黑话。逐个认识一下。

## ① 提示链：流水线工位

<Term id="prompt-chaining">提示链</Term>的结构最朴素：**固定顺序的多次调用，每步输出喂给下一步**，步与步之间还可以加「检查闸门」——中间产物不合格就提前拦截。

- **类比**：工厂流水线，每个工位只拧一颗螺丝。
- **例子**：先从文档提取要点 → 闸门检查要点数量 → 据要点写摘要 → 把摘要翻译成英文。
- **为什么有效**：每一步都比整体任务简单，模型在简单任务上更稳（[A2.1](../02-planning-reflection/01-planning.mdx) 的任务分解，这里是人替模型拆好了）。
- **何时失灵**：步骤其实互相依赖、切不干净——比如「翻译」需要回头改「摘要」的措辞时，硬切成两步反而两头不讨好。

## ② 路由：医院分诊台

<Term id="routing">路由</Term>是「先分诊，后处理」：**一次分类调用决定输入走哪条专门通道**，每条通道用自己的提示词、工具甚至模型。

- **类比**：医院分诊台——头疼去内科、骨折去骨科，没人让一位全科医生什么都看。
- **例子**：客服消息先分成「退款 / 报障 / 闲聊」，各走各的话术与流程；简单问题还可以路由给便宜的小模型，难题才上旗舰模型（2025 年常见的省钱配置）。
- **何时失灵**：类别边界模糊，分诊器自己都拿不准——错误的分诊比不分诊更糟，因为下游的专门通道对「走错门」的输入毫无准备。

## ③ 并行：分工阅卷与三个医生会诊

<Term id="parallelization">并行</Term>有两副面孔：

- **分片**：把任务切成独立的几块同时做——像十个老师分工阅卷，各改各的题；
- **投票**：同一个任务让多路独立做，再汇总或表决——像三个医生分别看片子再会诊，谁也不受谁影响。

例子：一段代码同时交给「安全、性能、风格」三个视角各查一遍再合并（分片）；一道数学题采样五次答案取多数（投票——上篇 [8.4 推理模型](/docs/frontier/reasoning-models)提过的 self-consistency 就是它）。

**何时失灵**：子任务其实有依赖（后一块要用前一块的结论，切开就断了线索）；或者「汇总」比子任务本身还难——三份意见互相打架时，合并器成了新瓶颈。

## ④ 编排者-工人：项目经理带队

<Term id="orchestrator-worker">编排者-工人</Term>是五块积木里唯一带「运行时决策」的：**一个中枢模型现场分析任务、动态拆出子任务清单、派给工人、最后汇总成品**。

- **类比**：项目经理接了个新项目，先开会拆活，再派给组员——每个项目拆法都不一样。
- **和并行的关键区别**：并行的「三块」是写代码时就定死的；编排者的子任务是**看了任务才知道**要拆成几块、拆成什么块。
- **例子**：写一份行业调研报告——具体查哪几个子方向，得先看初步资料才能定。
- **何时失灵**：编排者拆得烂——子任务重叠、遗漏、粒度失当，工人再强也白搭。中枢的规划质量就是全系统的天花板（[A2.1](../02-planning-reflection/01-planning.mdx) 的老话：验收标准要独立于计划）。

## ⑤ 评估者-优化者：作家与编辑

<Term id="evaluator-optimizer">评估者-优化者</Term>把 [A2.2 的<Term id="reflection">反思</Term>](../02-planning-reflection/02-reflection.mdx)拆成双角色：**一个负责生成，一个负责挑毛病，带着意见循环到达标为止**。

- **类比**：作家写稿、编辑退稿——各司其职，比作家自己闷头改更容易看见问题。
- **例子**：翻译初稿 → 审校按「信达雅」打分 → 不到 90 分带意见重写；生成代码 → 跑测试 → 报错回炉。
- **何时失灵**：A2.2 讲过的老问题——没有清晰的「及格线」，或者迭代根本不会变好。评估标准含糊时，这个循环就是在烧 token 原地转圈。

:::caution 这套分类在哪里不准确？
五模式是从实践中**提炼**出来的归纳，不是法律。真实系统常常是变体和混血：带重试的链、会降级的路由、编排者手下挂着评估循环……对不上「标准形状」很正常。更重要的告诫是方向：**先看任务形状，再选积木**——拿着刚学会的锤子找钉子、给简单任务套五层结构，是学完本节最容易犯的病，名字叫过度设计。
:::

<DeepDive title="五模式的实现骨架">

五块积木的代码都出奇地短——这正是 2025 年后业界渐渐少用重型编排框架、更多直接手写的原因（积木很薄，库反而重）：

```python
# ① 提示链：顺序 for + 中间校验
def chain(doc):
    points = llm(f"提取要点：{doc}")
    if count(points) < 3:                 # 检查闸门
        raise Retry("要点太少，换个切法重来")
    summary = llm(f"按要点写摘要：{points}")
    return llm(f"翻译成英文：{summary}")

# ② 路由：一次分类调用 + 分支表
def route(msg):
    kind = llm(f"分类（退款/报障/闲聊）：{msg}")
    return HANDLERS[kind](msg)            # 各分支自带提示词/工具/模型

# ③ 并行：并发调用 + 聚合
def parallel(code):
    reviews = run_concurrently([
        lambda: llm(f"只查安全问题：{code}"),
        lambda: llm(f"只查性能问题：{code}"),
        lambda: llm(f"只查代码风格：{code}"),
    ])
    return llm(f"合并去重这三份评审：{reviews}")

# ④ 编排者-工人：规划调用产出子任务，动态派发
def orchestrate(task):
    subtasks = llm(f"把任务拆成子任务清单（JSON）：{task}")
    results = run_concurrently([worker(t) for t in subtasks])
    return llm(f"汇总成最终交付：{results}")

# ⑤ 评估者-优化者：生成/批评循环
def refine(task, max_rounds=3):
    draft, feedback = llm(task), None
    for _ in range(max_rounds):
        feedback = llm(f"按标准挑毛病，达标输出 PASS：{draft}")
        if "PASS" in feedback:
            return draft
        draft = llm(f"按意见修改：{draft}\n意见:{feedback}")
    return draft                          # 超轮次：带瑕疵交付或转人工

```

**失败处理是积木的一部分。** 每一步模型调用都可能超时、输出格式错、内容不合格——重试（同参数再来）、降级（换简单方案或小模型）、上抛（转人工）三条路要在写积木时就选好，而不是上线后打补丁。②的分诊拿不准时给「兜底通道」、④的工人失败时编排者要能重派，都是同一件事。

**与编排框架的关系**（业界现象，2023-2026）：2023 年的链式框架把②③④做成了库和 DSL；随着大家发现每块积木不过十行代码、调试时反而被框架的抽象层挡住视线，2025 年后「手写积木 + 只借框架的观测与重试设施」成了越来越常见的选择。框架没有消失，是回到了合适的位置。

</DeepDive>

## 小结

:::tip 本节要点
- 五种积木：提示链（固定串行）、路由（先分诊）、并行（分片或投票）、编排者-工人（运行时现拆）、评估者-优化者（生成 + 挑毛病循环）。
- 前三种的步骤在写代码时定死；编排者-工人是唯一把「拆任务」交给模型现场决定的积木。
- 每种积木都有失灵条件——选型先看任务形状，别拿着锤子找钉子。
:::

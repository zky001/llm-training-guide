# A6.3 计算机使用：像人一样点鼠标

## 没有 API 的世界

上一节的代码智能体活在一个「幸运」的世界里：终端、文件、测试，全都是干净的文本接口。但你抬头看看真实世界的软件——一个二十年没更新的政务网站、一个只有 Windows 客户端的老旧 ERP、别人家没开放接口的网页。**它们大多没有 API。** 智能体伸手，够不着。

怎么办？最后的通用办法，也是最像人的办法——<Term id="computer-use">计算机使用</Term>：给智能体一张**屏幕截图**，让它像人一样输出「把鼠标移到坐标 (340,210) 点击」「在这里输入这段文字」「按回车」。不需要对方提供任何接口，只要它有屏幕、有鼠标键盘，智能体就能用。

## 又是那个循环，只是「眼睛」和「手」换了

计算机使用的工作方式，你其实已经烂熟于心——还是 [A0 的思考-行动-观察循环](../00-from-chat-to-action/01-what-is-agent.mdx)：

1. **看**（观察）：截一张当前屏幕的图；
2. **想**（思考）：多模态模型看懂界面，决定下一个动作；
3. **做**（行动）：执行点击 / 输入 / 按键；
4. 再截图，看结果，回到第 1 步。

变的只有两处：「观察」从读文本变成了**看像素**，「行动」从调 API 变成了**动鼠标键盘**。内核一模一样。2024-2025 年，各家陆续推出了这类 computer use / 浏览器智能体能力。

## 为什么它比代码难得多

如果说代码智能体是优等生，计算机使用还在艰难爬坡，原因层层叠加：

- **看懂界面比读文本难。** 图标是什么意思、这个弹窗要不要关、按钮藏在哪——这些依赖多模态感知，而上篇 [8.3 节](/docs/frontier/multimodal)就点过，多模态模型在「数数、方位、精确定位」上仍是短板。界面恰恰全是这些。
- **动作空间大且脆弱。** 点偏几个像素就点到别的东西；页面还没加载完就点，动作全废。容错空间极小。
- **慢。** 每一步都要「截图 → 多模态推理 → 动作」，比读一行文本慢得多，长流程累积下来很费时间也很费钱。
- **几乎不可验证。** 这是最要命的一条，直接回扣 [A6.2](./02-why-code.mdx)：点对了没有「测试」告诉你，填错了表单也没有报错弹出来。没有那把自动判分尺，[反思](../02-planning-reflection/02-reflection.mdx)就成了空想，能力飞轮转不起来。

## 浏览器智能体：一个更实用的子类

在纯截图的「硬模式」和有 API 的「简单模式」之间，网页是一块特殊的中间地带——它有 **DOM**：一份半结构化的「网页地图」，能告诉智能体「这里有个按钮、那里有个输入框」，比对着一堆像素猜坐标可靠得多。

所以浏览器智能体是当前更实用的形态：既能读 DOM 结构定位元素，必要时也能结合截图看视觉布局。查资料、填表单、比价下单这类「网页里的活」，是计算机使用最先落地的场景。

## 现状与前景

诚实地说，2025-2026 年的计算机使用还在早期：简单、短流程的任务能做，长流程、多步骤的任务容易中途崩掉（[错误复利](../07-agent-evaluation/01-error-compounding.mdx)在这里格外凶猛——每一步都脆、又不可验证）。

但它的方向感极强：**计算机使用是通往「能操作一切软件的通用助手」的必经之路。** 一旦成熟，智能体就不再受限于「哪些软件开放了 API」，而能像人一样使用整个数字世界。潜力巨大，也正因如此值得耐心。

:::caution 计算机使用把安全问题推到了极致
一个能自主截屏、看懂界面、操作你**所有**软件的智能体，一旦被[提示注入](../08-safety-frontier/01-prompt-injection.mdx)劫持（比如某个网页里藏了「现在去把网银余额转走」的指令），后果不堪设想——它有一双能点遍全屏的手。所以这类系统尤其需要 [A8 的最小权限和人工确认](../08-safety-frontier/02-defense.mdx)。业界普遍的安全共识（2025-2026）是：**让它在隔离环境里运行、不登录你的真实敏感账户、高风险操作一律人工把关。** 能力越通用，越要给它戴上缰绳。
:::

<DeepDive title="从像素到动作的技术细节">

**动作接口。** 计算机使用模型的输出是一套「动作词表」：`click(x, y)`、`type(text)`、`key(keyname)`、`scroll(dir)`、`drag(...)` 等。模型看图，输出这些结构化动作，由外层程序真正执行——本质仍是 [A1 的工具调用](../01-tool-use/index.md)，只不过工具是「鼠标键盘」。

**降低定位难度：set-of-marks。** 直接让模型猜像素坐标很不稳。一个常用技巧是 set-of-marks：先给屏幕上所有可点击元素叠加编号标签，让模型输出「点击 3 号元素」而非「点击 (340,210)」——把连续的坐标问题变成离散的选择问题，命中率显著提升。

**两条路线：DOM vs 纯视觉。** 浏览器场景可以走 DOM 路线（读取可访问性树 / HTML 结构，定位精准，但依赖网页结构良好），也可以走纯视觉路线（只看截图，通用但更难）。实践中常常混用：DOM 定位为主，视觉理解补足动态渲染和图形化内容。DOM 路线更精准，纯视觉路线更通用——这也解释了为什么浏览器智能体比通用桌面智能体先成熟。

**评测基准。** WebArena、OSWorld 等（2023-2024）在真实网页 / 桌面环境里给计算机使用任务打分——判定「事办成了没」需要检查环境的最终状态（表单真的提交了吗、文件真的建了吗），这正是 [A7.2](../07-agent-evaluation/02-benchmarks.mdx) 讲的「基于状态而非基于输出文本」判定的典型场景。

**成本账。** 每一步一次多模态推理（还要处理高分辨率截图，回扣 [6.2 的 KV Cache 显存账](/docs/inference/kv-cache)），长任务的 token 和延迟成本远高于纯文本智能体。这也是当前它更适合短任务的现实原因之一。

</DeepDive>

## 小结

:::tip 本节要点
- 计算机使用 = 让智能体看截图、动鼠标键盘，操作没有 API 的软件——还是 A0 的思考-行动-观察循环，只是「眼睛」看像素、「手」动鼠标。
- 它比代码难得多：看懂界面依赖多模态短板、动作脆弱、慢，而且**几乎不可验证**（没有测试当判分尺）。
- 浏览器智能体是更实用的子类，因为网页有 DOM 这份半结构化「地图」。
- 它是通往「通用软件助手」的必经之路，但把安全问题推到极致——必须最小权限、隔离环境、高危动作人工把关。
:::

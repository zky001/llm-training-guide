# A5.1 为什么要多个：三笔真实的账

## 一个人能干的活，为什么要雇一个团队？

现实中的公司从不用「人多力量大」这种虚话回答这个问题。真正的理由永远算得清：一个人脑子装不下全部细节（所以分部门）、十件事串行做太慢（所以并行开工）、自己查自己查不出毛病（所以设独立审计）。

多智能体系统的账本一模一样。把 [A0 那个「模型 + 工具 + 循环」](../00-from-chat-to-action/01-what-is-agent.mdx)的<Term id="agent">智能体</Term>复制几份、让它们分工协作，值不值，取决于下面三笔账里你能不能占到便宜。

## 账①：上下文隔离——最硬的一笔

[A3.1 章](../03-memory-context/01-context-window.mdx)反复敲打过：上下文是稀缺资源，塞得越满质量越差。

现在想象一个任务：「调研 10 个候选方案，写一份对比报告」。单智能体做，10 份调研资料全部灌进同一条上下文——查到第 7 个方案时，窗口已经塞满前 6 个方案的搜索结果和中间笔记，早期信息开始腐烂，后期调研开始敷衍。

换成 10 个<Term id="sub-agent">子智能体</Term>：每人只调研一个方案，中间过程（搜索结果、比价笔记、失败的查询）**全部留在自己的上下文里消化**，最后只把三五百 token 的结论交回来。主智能体的上下文里永远只有干净的结论，没有过程垃圾。这就是<Term id="context-isolation">上下文隔离</Term>——多智能体最硬的一笔收益：**它买到的不是「更多算力」，而是「更多干净的工作记忆」。**

## 账②：并行——墙钟时间只算最慢的

10 个子方向逐个查，总时间是 10 份之和；10 个子智能体同时查，总时间约等于**最慢的那一个**。

这个直觉你在上篇见过：[4.4 章的并行训练](/docs/pretraining/parallelism)把计算切给几千张 GPU 分摊。这里只是把「多卡分摊计算」换成了「多智能体分摊任务」——对调研、审计、批量处理这类天然可拆的任务，加速几乎是白捡的。

## 账③：角色专业化与对抗——审稿人就该挑刺

[A2.2 的 caution](../02-planning-reflection/02-reflection.mdx) 留过一个尾巴：让模型自己反思自己，「评估者和生成者偏差共享」，盲区照不亮。

多智能体给出解法：**不同的系统提示词 = 不同的人格**。给写手的提示词说「高产、大胆、先完成再完美」，给审稿人的说「你的唯一职责是挑毛病，放过一个错误就是失职」——同一个底座模型，装进两个对立的角色里，比一个角色内心纠结要有效得多。挑刺的没有「护犊子」的动机，生成的不必畏首畏尾。

## 反面账单：多出来的三笔支出

天下没有免费的组织架构。多智能体同时欠下三笔债：

| 支出 | 为什么 |
| --- | --- |
| 总 token 更多 | 每个子智能体都要重新读一遍任务背景和自己的说明书——背景被复制了 N 份 |
| 消息必丢信息 | 工人 6000 token 的调研过程压成 500 token 结论——工人看到的 ≠ 主智能体知道的 |
| 协调成本 | 两个工人的结论打架，谁裁决？裁决依据哪来？这些在单智能体里根本不存在 |

:::info 决策口诀
**任务能被一条上下文舒服装下 → 单智能体。** 装不下、或天然可并行、或需要对抗视角 → 才考虑多智能体。这是 [A0.2「能简单就简单」](../00-from-chat-to-action/02-autonomy-spectrum.mdx)在更高一层的重演。
:::

:::caution 行业教训（2025-2026）
多智能体框架热潮里出现过各种花哨拓扑：全连接聊天室、自由辩论、无中枢集市……生产实践给出的答案相当保守——**这些结构大多打不过朴素的「编排者-工人」**。结构越自由，失败模式越多：话题漂移、循环扯皮、责任真空。业界共识可以概括成一句话：不要造聊天室，要造流水线。下一节展开。
:::

<DeepDive title="多智能体的成本模型">

**token 账。** 设任务背景 b、每份子任务的工作量 w、子任务数 N。粗账：

- 单智能体：约 `b + N·w`（背景只读一遍，但全挤在一条上下文里）
- 多智能体：约 `N·(b' + w) + 编排开销`（b' 是每个工人拿到的任务书，背景被部分复制 N 份）

注意结论：多智能体的**总 token 几乎总是更多**。它省的从来不是总量，而是**单条上下文的峰值占用**——把一条 15 万 token 的臃肿上下文，换成 N 条 1~2 万的清爽上下文。按 [A3.1](../03-memory-context/01-context-window.mdx) 的「有效窗口远小于标称窗口」，这笔交换通常划算。

**通信即压缩。** 工人交回结论的那一刻，发生了一次有损压缩：6000 token 的过程 → 500 token 的结论。系统能不能工作，取决于**丢掉的是不是真没用**。所以结论模板（下一节）要强制工人上报「依据、置信度、未尽事项」——把最容易被压缩掉、又最影响决策的东西显式保下来。

**何时并行不划算。** 并行的前提是子任务独立。若子任务有依赖链（B 要用 A 的产出），并行度退化为 1，多智能体只剩通信与协调的纯支出。判断方法很朴素：把子任务画成依赖图，看最长链占总工作量的比例——比例越高，越该用单智能体顺序做。

</DeepDive>

## 小结

:::tip 本节要点
- 多智能体的三笔收益：上下文隔离（最硬）、并行提速、角色专业化与对抗。
- 三笔代价：总 token 更多、消息传递必丢信息、协调成本凭空出现。
- 口诀：一条上下文装得下就别拆；隔离买的是「干净的工作记忆」，不是「更多算力」。
:::

# A7.2 智能体 Benchmark：怎么给「会办事」打分

## 从「答得对」到「办成了」

上篇[第 7 章](/docs/evaluation/benchmarks)讲过怎么给模型打分：出一道题，看它答得对不对。给智能体打分完全是另一回事——你不是在问它「北京到上海怎么订票」，而是让它**真的去订**，然后检查：票订上了吗？日期对吗？没超预算吧？

难点一下就冒出来了：要评一个智能体，你得先搭一个**能自动判定「成功」的真实环境**。这比出一道选择题难太多了。

## 主流智能体 Benchmark 一览

看看业界是怎么解决「判成功」这个难题的（各标提出年份）：

| Benchmark | 考什么 | 环境 | 怎么判成功 | 一句点评 |
| --- | --- | --- | --- | --- |
| **SWE-bench**（2023） | 修真实的 GitHub bug | 真实开源仓库 | 跑官方测试套件，全绿即通过 | 判据最硬核——测试是客观的，回扣 [A6](../06-computer-use/index.md) |
| **GAIA**（2023） | 需要多工具协作的助理任务 | 网页+文件+计算 | 对照唯一正确答案 | 逼智能体串起搜索、读文档、算数 |
| **WebArena / OSWorld**（2023/2024） | 在真实网页/桌面上操作 | 可复位的真实软件环境 | 检查最终状态（订单建了吗、文件改对了吗） | 环境即评测，回扣 [A6.3 计算机使用](../06-computer-use/index.md) |
| **τ-bench**（2024） | 模拟客服（订票、退货） | 带数据库的对话环境 | 检查数据库最终状态 + pass^k 稳定性 | 把[稳定性](./01-error-compounding.mdx)当一等公民 |

一个共同点：越靠谱的 benchmark，越是**用「环境的最终状态」而不是「模型说了什么」来判成功**。SWE-bench 不看智能体如何雄辩地解释自己的修复，只看测试过不过；τ-bench 不看它多会安慰用户，只看数据库里退款到底退没退。

## 智能体评测的三个特有难点

比起模型评测，智能体评测多了三座大山：

1. **判成功难**。开放任务往往没有唯一正解（「写一份调研报告」怎么算对？）。要么搭一个能自动判状态的真实环境（贵），要么请 <Term id="llm-as-judge">模型当裁判</Term>（继承了上篇 [7.2](/docs/evaluation/pitfalls) 讲的长度、自信、位置各种偏差）。
2. **轨迹也要评**。不能只看结果对不对，还要看**过程**：它有没有绕远路烧掉十倍 token？有没有偷偷执行了危险操作？有没有把用户数据发到不该发的地方（[A8 的正题](../08-safety-frontier/index.md)）？结果对了、过程惊悚，一样不能上线。
3. **贵且慢**。评一道选择题是一次模型调用；评一个智能体任务是一整轮多步交互，可能几十次调用、跑几分钟。跑一遍完整评测集的成本，是模型评测的几个数量级以上。

## 老坑一个没少，还多了新的

上篇 [7.2](/docs/evaluation/pitfalls) 讲的评测陷阱，在智能体时代全部继承：

- <Term id="data-contamination">数据污染</Term>：SWE-bench 里的知名 issue 和它的修复，早就在训练数据里躺着了——模型可能是「背过这道题」而不是「会修 bug」。
- 刷榜：针对某个 benchmark 的特定环境、特定脚手架过拟合，分数好看，换个场景就废。

而且还多了一个新坑：智能体的分数**高度依赖脚手架**（工具集、提示词、上下文管理策略）。同一个模型，配一套好工具和配一套烂工具，SWE-bench 分数能差出一大截——所以「某模型 SWE-bench 多少分」这个说法本身就不完整，得问「用的什么智能体框架」。

:::caution 榜单分数 ≠ 你的场景表现（这次更严重）
模型评测里「榜单分不代表实际体验」已经够坑了，智能体这里更甚：智能体的表现是**模型 × 工具 × 提示词 × 上下文策略**的乘积，换任何一个因子结果都可能天翻地覆。「SWE-bench 排第一」离「能修好你公司代码库的 bug」差着十万八千里——你的代码、你的工具、你的任务分布，都不在那个榜单里。**唯一靠谱的评测，是在你自己的任务上评测**。这直接引出下一节。
:::

<DeepDive title="搭一个智能体评测环境">

**环境即评测。** 智能体评测的核心不是「一道题」，而是「一个可复位的沙箱 + 一条成功判据」。跑评测 = 把智能体扔进沙箱、让它折腾、然后检查沙箱的最终状态。

**判据要基于状态，不要基于文本。** 这是最容易踩的坑：

- ❌ 基于输出：「智能体说它订好票了」——它可能在撒谎或搞错了。
- ✅ 基于状态：查订票系统的数据库，`2026-07-20 北京→上海` 这条订单真的存在吗、金额对吗。

同理，测「改文件」看文件内容和测试，测「建订单」看数据库记录。让**世界的真实变化**来判成功，而不是让模型自己汇报。

**LLM-as-judge 用在智能体上。** 对没法用状态判定的开放任务（报告质量、回复得体程度），只能上模型裁判。它继承了上篇 [7.2](/docs/evaluation/pitfalls) 的全部偏差（长度、自信、位置、自恋），缓解手段也一样：交换位置取平均、多裁判集成、按维度分别打分。能用状态判的地方，永远优先用状态判。

**轨迹级指标。** 除了成功率，成熟的评测还盯：

| 指标 | 照出什么问题 |
| --- | --- |
| 平均步数 / token | 有没有绕远路、烧钱 |
| 工具错误率 | 工具选错、参数填错的频率（回扣 [A1](../01-tool-use/index.md)） |
| 是否触碰危险动作 | 有没有执行未授权的删除、发送（[A8](../08-safety-frontier/index.md)） |
| pass^k | 稳定性（[A7.1](./01-error-compounding.mdx)） |

**离线回放 vs 在线交互。** 离线回放（拿历史轨迹重跑某一步）便宜但覆盖不到「环境会因智能体的动作而变化」的情况；在线交互（真的让它跑完整任务）真实但贵。生产中两者搭配用。

</DeepDive>

## 小结

:::tip 本节要点
- 给智能体打分 = 搭一个能自动判定「事办成没」的真实环境，判据要基于**状态**不是基于**输出文本**。
- 主流 benchmark：SWE-bench（跑测试）、GAIA（多工具助理）、WebArena/OSWorld（真实软件操作）、τ-bench（客服+稳定性）。
- 智能体评测三难：判成功难、轨迹也要评、贵且慢；污染和刷榜的老坑一个没少。
- 分数高度依赖脚手架，榜单第一 ≠ 你的场景能用——一定要在自己的任务上评测。
:::

# A1.1 Function Calling：一次「伸手」的全流程

## 模型没有手

先戳破一个想象：模型并不能「运行」任何东西。它没有网络权限、没有文件系统、碰不到你的日历——它从头到尾只会一件事：**生成文本**（上篇讲了九章的那件事）。

那它是怎么「调用」计算器、查到天气的？答案巧妙得近乎狡猾：

> **它生成一段描述「我想调用什么」的文本，真正动手的是旁边的普通程序。**

就像一位被锁在房间里的军师：他不能亲自上阵，但他可以往门缝外递纸条——「派骑兵去左翼」。只要门外有个传令兵照着纸条执行、再把战报塞回来，军师就等于「指挥」了整场战役。<Term id="function-calling">工具调用</Term>（Function Calling）就是这套递纸条的规矩。

## 全流程五步

一次完整的「伸手」是这样的：

1. **上清单**：开发者把工具清单放进上下文。每个工具带一份<Term id="tool-schema">工具描述</Term>——名字、用途、参数格式（JSON Schema），相当于说明书；
2. **做决策**：模型读用户的问题、读说明书，判断「回答这个问题，缺不缺外部信息或外部操作」——缺，才需要工具；
3. **递纸条**：需要就生成一段结构化调用请求（JSON）：调哪个工具、参数填什么。注意这一步**仍然是生成文本**，只是格式特殊；
4. **传令兵执行**：外层程序解析这段 JSON，真正去执行（查 API、跑计算），把结果拼回上下文；
5. **继续生成**：模型读到结果，要么组织成人话回答，要么再递一张纸条——这就转起了 [A0 章](../00-from-chat-to-action/01-what-is-agent.mdx)的<Term id="agent-loop">行动循环</Term>。

**建议的玩法**：按顺序玩五个问题——算术 → 天气 → 汇率 → 日程 → 寒暄。前四个各演示一类「模型自身做不到的事」（精确计算、实时信息、变动数据、私人数据）；最后的寒暄题会告诉你，**判断「不需要工具」同样是正确决策**。然后勾上「⚠️ 把工具描述换成含糊版」，重玩汇率题和日程题：看着模型拿着「今天下午三点我有空吗」去搜索互联网，你就永远忘不了下面这一段。最后再勾上「🔧 注入工具执行失败」，重玩天气题或汇率题——工具会先返回一句报错（日期格式不对、币种要用 ISO 代码），看模型**读懂报错、自己改对参数重试**。这正是本节深入层要讲的「错误处理」在真实智能体里每天发生的样子。

## 描述即接口

模型决策时能看到的只有两样东西：用户的问题，和你写的工具描述。它没法打开工具「看源码」——**描述写成什么样，工具在模型眼里就是什么样**。

刚才的实验里，把 `exchange_rate` 的描述从「按实时汇率换算两种货币的金额」换成「处理数字」，模型就转头选了听起来更像「处理数字」的 `calculator`，用记忆里的旧汇率硬算，差了 200 多块。同一个模型、同一个问题、同样的工具代码——只是说明书烂了，任务就砸了。

所以工程圈有个略显扎心的共识：**写工具时最值钱的不是代码，是那一句描述。** 好的工具描述像好的 API 文档：说清楚它能干什么、不能干什么、参数的含义和格式，甚至给一两个使用示例。

:::caution 这个演示在哪里不准确？
组件里的「模型决策」是规则模拟——为了教学，它整齐得像查表。真实模型的工具选择是训练出来的「直觉」（下一节细讲），没有如此清晰的规则，也因此会以更意外的方式出错。真实系统还要兜底三件事：模型**幻觉出清单里不存在的工具**、参数 JSON 格式不合法、工具本身执行失败——这些错误处理占了工具调用工程的大半工作量（见深入层）。
:::

<DeepDive title="工具调用的报文与工程细节">

**真实的报文长什么样。** 以主流 API 的风格为例（各家格式略有差异且随版本演化，以官方文档为准，2025 年）。请求时带上工具清单：

```json
{
  "messages": [{"role": "user", "content": "899 美元大概是多少人民币？"}],
  "tools": [{
    "name": "exchange_rate",
    "description": "按实时汇率换算两种货币的金额",
    "input_schema": {
      "type": "object",
      "properties": {
        "amount": {"type": "number", "description": "金额"},
        "from": {"type": "string", "description": "源币种，如 USD"},
        "to": {"type": "string", "description": "目标币种，如 CNY"}
      },
      "required": ["amount", "from", "to"]
    }
  }]
}
```

模型的回复里出现一个工具调用块：

```json
{
  "role": "assistant",
  "content": [{
    "type": "tool_use",
    "id": "call_01",
    "name": "exchange_rate",
    "input": {"amount": 899, "from": "USD", "to": "CNY"}
  }]
}
```

开发者执行后，把结果作为一条新消息发回去：

```json
{
  "role": "user",
  "content": [{
    "type": "tool_result",
    "tool_use_id": "call_01",
    "content": "{\"value\": 6421.6, \"rate\": 7.14}"
  }]
}
```

模型读到这条 `tool_result`，继续生成最终回答。整个「调用」就是三条消息的接力。

**怎么保证输出一定是合法 JSON？** 靠约束解码（constrained decoding / grammar）：在采样每个 token 时（上篇 [3.6 节](/docs/transformer/sampling)的那个采样），把「会导致 JSON 非法」的候选直接屏蔽掉——语法上走不通的路，概率再高也不许走。这把「格式错误」从概率问题变成了结构问题。

**并行工具调用。** 现代模型可以一轮递出多张纸条（比如同时查北京和上海的天气），外层并行执行、一起回填——省下串行等待的轮次。

**错误处理三板斧。** ①瞬时失败（网络抖动）：外层直接重试；②模型的错（参数不合法、调了不存在的工具）：把报错信息作为 `tool_result` 回填，让模型自己看着错误改——它通常真能改对；③反复失败：降级，让模型明确告知用户「这一步没办成」，绝不假装成功。

**工具数量的代价。** 每个工具的说明书都要塞进上下文：几个工具无所谓，几十上百个工具会吃掉大量 token、还会稀释模型的注意力（选择困难是真实存在的）。算个量级：一个工具的 JSON schema（名字 + 描述 + 参数定义）通常占 **100~300 token**，50 个工具就是**上万 token 常驻**在每一次调用的上下文里——既按量付费地烧钱，又把模型的注意力摊薄到几十份说明书上，反而更容易选错。怎么办？按需加载工具、分层路由——这是 A3 章上下文工程和 A5 章 MCP 协议的伏笔。

</DeepDive>

## 小结

:::tip 本节要点
- 模型没有手：「调用工具」= 生成一段结构化的调用请求文本，真正执行的是外部程序。
- 五步流程：上清单 → 做决策 → 递纸条（生成 JSON）→ 外部执行 → 结果回填继续生成。
- 描述即接口：模型眼里的工具就是你写的那句描述，说明书的质量直接决定调用的对错。
:::

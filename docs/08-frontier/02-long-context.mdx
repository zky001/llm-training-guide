# 8.2 长上下文：从 4K 到百万 token

## 一场军备竞赛

2020 年的 GPT-3，<Term id="context-window">上下文窗口</Term>只有 2048 个 token——大约半篇论文，聊天稍长一点它就把开头忘了。到 2025 年，旗舰模型的窗口已经卷到**百万级**：一次塞进几本书、整个代码仓库、几小时的会议记录。

五年一千倍。这不是把一个数字改大那么简单——每一步都在跟三笔账单搏斗。

## 为什么长上下文这么难

**第一笔：计算的平方账。** [3.1 节深入层](../03-transformer/01-attention-intuition.mdx)埋过一个伏笔：注意力要让每个 token 和其他所有 token 打交道，计算量是 $O(n^2)$——现在正式兑现。上下文从 4K 涨到 400K，长度涨 100 倍，注意力计算涨 **10000 倍**。

**第二笔：显存的线性账。** [6.2 节](../06-inference/02-kv-cache.mdx)算过：<Term id="kv-cache">KV Cache</Term> 随上下文长度线性膨胀。百万 token 的缓存动辄几十上百 GB——比模型本身还大，一张卡直接被「记忆」撑爆。

**第三笔：训练的分布账。** 语料里超长文档本来就稀缺（互联网上大多是短文），而且模型在 4K 长度上训练后，[3.3 节](../03-transformer/03-multihead-position.mdx)的位置编码没见过第 100 万号「工牌」——直接外推会失灵。好在 RoPE 有个漂亮的性质：把旋转的「基频」调慢（相当于把钟表指针放慢，同样的表盘覆盖更长的时间），配合少量长文本继续训练，就能把窗口拉长好几倍。

## 工程组合拳

实际的长上下文方案是一套组合拳，而且几乎每一招你都在前面见过原理：**稀疏/滑动窗口注意力**砍平方账（不是每个 token 都需要看所有人）、**RoPE 频率缩放**解决位置外推、**GQA（分组查询注意力）**给 KV Cache 瘦身（[6.4 节](../06-inference/04-serving.mdx)提过）、再配上 [4.4 节](../04-pretraining/04-parallelism.mdx)家族新增的序列并行把超长序列切到多卡上训练。

## 「大海捞针」：长了，但读懂了吗

标称一百万，实际行不行？业界流行的体检叫**大海捞针**（needle in a haystack，2023 年起流行）：在超长文本的任意位置藏一句无关的话（「小明最喜欢的颜色是蓝色」），看模型能否准确捞出。2024 年后的旗舰模型在这项测试上普遍接近满分——但别高兴太早：**捞针只考「查找」**。要求模型对百万 token 做真正的综合推理（对比两章观点、追踪一条跨越全书的线索）时，成绩仍明显掉队（2024-2025 年多项长文推理评测的共同结论）。

另外常有人问：有了百万上下文，还需要外接检索（RAG）吗？目前的共识是**互补**：检索负责从海量库里粗筛，长上下文负责把筛出的材料一次读透——一个是图书馆的索引卡，一个是更大的书桌。长上下文并没有取代 RAG，两者是搭档——这笔账（成本、窗口上限、精度）本站[中篇 K7.3「RAG vs 长上下文」](/kb/frontier/rag-vs-long-context)用一个计算器算给你看。

:::caution 这个「百万」在哪里不准确？
厂商标称的「支持 X token」只承诺**能塞进去**，不承诺**在 X 长度下依然聪明**。有效上下文（性能不明显衰减的长度）常常显著短于标称值，而且不同任务衰减速度不同。看长上下文能力，别只看一个数字，要看「长度-性能」曲线（2025 年的评测惯例）。
:::

<DeepDive title="打破平方的尝试">

**滑动窗口注意力**。每个 token 只看最近 $w$ 个 token（Mistral-7B，2023，$w=4096$），计算量降为 $O(nw)$。远处信息靠层层传递间接到达——层数越深，「感受野」越大，代价是远距离信息会衰减。

**稀疏模式**。滑窗之外再保留少数「全局 token」（如开头几个、或按步长抽样），兼顾局部细节与全局锚点。各家的稀疏花样不同，思想一致：绝大多数 token 对不重要，别在它们身上花 $O(n^2)$。

**换掉注意力的激进路线**。线性注意力与状态空间模型（Mamba，2023）把复杂度压到 $O(n)$：不保存全部历史，而是把历史压缩进固定大小的「状态」。代价是精确检索能力变弱（状态是有损压缩），截至 2026 年初，纯线性路线尚未撼动主流，混合架构（多数层滑窗 + 少数层全注意力）反而成了实用解。

**RoPE 频率缩放**。RoPE 用不同频率的「旋转」编码位置（[3.3 节](../03-transformer/03-multihead-position.mdx)的钟表指针类比）。把基频整体调慢 $k$ 倍，原本覆盖 4K 的表盘就能覆盖 $4K \times k$——位置间的相对关系近似保持，模型只需少量长文本微调即可适应（NTK/YaRN 系方法，2023）。

**捞针评测的局限**。针是「与背景无关的异质信息」，天然显眼；真实需求里要找的往往是与上下文高度相似的内容。满分捞针 ≠ 长文理解，多跳版（针 A 指向针 B）与聚合版（统计全文出现次数）评测更接近真实难度，模型成绩也惨得多（2024-2025）。

</DeepDive>

:::tip 本节要点
- 长上下文的三大敌人：注意力的 $O(n^2)$ 计算、KV Cache 的线性显存、位置编码的外推失灵。
- 解法是组合拳：滑窗/稀疏注意力、RoPE 频率缩放、GQA 瘦身缓存、序列并行。
- 大海捞针满分只代表「查得到」，长文综合推理仍是普遍短板（2025）。
- 标称窗口 ≠ 有效窗口，看长度-性能曲线才算数。
:::

# 8.1 MoE：万亿参数，百亿账单

## 一个诱人的问题

[4.3 节](../04-pretraining/03-scaling-laws.mdx)的 <Term id="scaling-laws">Scaling Laws</Term> 说：参数越多，模型越强。但稠密模型有个死规矩——**每个 token 都要动用全部参数**。参数翻十倍，每个 token 的计算量也翻十倍，训练和推理的账单一起爆炸。

有没有办法让参数无限涨、计算量却不涨？

有。回想 [3.5 节](../03-transformer/05-gpt-assembly.mdx)拆过的参数账本：Transformer 块里的<Term id="feed-forward">前馈网络</Term>占了参数的大头。<Term id="moe">MoE</Term> 的主意就打在它头上——

**把每层那一个前馈网络，换成一群「专家」（每个专家就是一个前馈网络）加一个「路由器」；每个 token 进来，路由器现场挑其中两位来处理，其余专家原地休息。**

多点几个 token，注意两件事：不同 token 会被派给**不同的专家组合**；无论派给谁，每次干活的都**只有 2 位**。底部的账本就是 MoE 的全部魔法：8 位专家各 7B、共享部分 3B，参数总量 59B——但每个 token 只激活 17B。**知识容量按 59B 算，算力账单按 17B 付。**

## 收益与代价

收益很直白：在几乎不增加计算量的前提下，把「脑容量」扩大几倍——语料里那些互不相干的知识（Python 语法、唐诗格律、法律条文）可以分散存进不同专家，不必挤在同一套参数里打架。

代价也很实在：

- **显存不打折**：计算可以只用 2 位专家，但显存必须装下**全部** 8 位——MoE 省算力，不省显存（回想 [4.4 节](../04-pretraining/04-parallelism.mdx)的显存账本，这又是一道要切分的难题）。
- **路由会偏心**：训练中路由器容易「用熟不用生」——热门专家挤爆、冷门专家躺平退化。所以要额外加一个负载均衡损失，逼它雨露均沾（细节见深入层）。

## 从边缘到主流

MoE 思想很老（1991 年就有），但在大模型时代的走红路线很清晰：2023 年 Mixtral 8×7B 开源，公开验证了「MoE 能打」；2024 年 DeepSeek-V3 做到 671B 总参数、每 token 只激活 37B——比本节实验的账本夸张得多；坊间长期传闻 GPT-4 也是 MoE 架构（**传闻，未获官方证实**）。到 2025 年，旗舰级开源模型已有大半采用 MoE。

:::caution 这个演示在哪里不准确？
实验里的「路由」是用固定规则模拟的，真实路由器是一个训练出来的小网络。更重要的误区是「专家」这个词本身：真实 MoE 的专家分工是训练**涌现**的，研究者检查后发现它们大多按 token 的表层特征（标点、数字、语种）分流，而不是按「Python 专家」「唐诗专家」这样的人类概念分工——别被名字骗了。
:::

<DeepDive title="路由与负载均衡">

**路由器的数学**。路由器就是一个线性层：对 token 向量 $x$ 打分 $s = \mathrm{softmax}(W_r x)$，得到它对每位专家的「亲和度」，取 top-$k$（常见 $k=1$ 或 $2$）位专家，输出为被选专家结果的加权和：

$$
y = \sum_{i \in \text{top-}k} s_i \cdot E_i(x)
$$

实验里「专家 4（63%）+ 专家 7（37%）」显示的就是归一化后的 $s_i$。

**负载均衡辅助损失**。若只优化语言建模损失，路由会塌缩到少数专家（强者恒强）。解法是加一项辅助损失，惩罚「分配比例偏离均匀」的程度——直觉上相当于给路由器一条纪律：一个批次里每位专家接到的 token 数应当大致相同。系数要调得小心：太强会牺牲路由质量，太弱则均衡失效。

**专家并行（EP）**。专家天然是可切分单元：把 8 位专家放到 8 张卡上，token 算到 MoE 层时路由到对应卡。这是 [4.4 节](../04-pretraining/04-parallelism.mdx)并行家族的新成员，代价是 all-to-all 通信（token 要跨卡「串门」）。

**细粒度与共享专家**。DeepSeek-MoE（2024）把少数大专家拆成许多小专家（提高组合自由度），并保留几位「共享专家」处理所有 token（存放通用知识），是 2025 年前后被广泛借鉴的设计。

</DeepDive>

:::tip 本节要点
- MoE 把前馈网络换成「专家群 + 路由器」，每个 token 只激活 top-2 专家。
- 参数容量与计算成本解耦：知识按总参数算，账单按激活参数付。
- 显存不打折（全部专家都得在卡上），路由要靠负载均衡损失防止偏心。
- 专家分工是训练涌现的，大多不对应人类可读的概念。
:::

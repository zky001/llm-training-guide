# 第 4 章 · 小结与自测

## 全章鸟瞰

把这一章串成一条生产线，就是「原料 → 预算 → 产线 → 长跑」：

```mermaid
flowchart LR
    A["原始网页<br/>(互联网大垃圾场)"] --> B["清洗流水线<br/>提取/过滤/去重/脱敏"]
    B --> C["配比混合<br/>网页+代码+书籍"]
    C --> D["定预算<br/>Scaling Laws: C≈6ND"]
    D --> E["3D 并行<br/>TP×PP×DP/ZeRO 上万卡"]
    E --> F["数月长跑<br/>混合精度+裁剪+容灾"]
    F --> G["base 模型"]
```

一段话说完本章：**互联网原始文本经过几十道清洗工序，只有百分之几能入库（4.1）；模型在这些文本上做的唯一一件事仍是预测下一个词，知识与推理是这个目标逼出来的副产品（4.2）；参数开多大、数据喂多少，Scaling Laws 让这笔几亿元的账可以提前算（4.3）；训练态显存 = 参数 × 16 字节，一张卡装不下，就用 TP、PP、DP/ZeRO 组合拳切开分摊（4.4）；最后靠混合精度、梯度裁剪和 checkpoint 容灾，扛住数月长跑里平均几小时一次的故障（4.5）。** 产出的 base 模型，就是下一章后训练的起点。

## 要点回顾

| 小节 | 两行记住它 |
| --- | --- |
| [4.1 数据从哪来](./01-data.mdx) | 互联网是垃圾场，清洗流水线决定模型上限；去重防「背题」，配比是秘方。 |
| [4.2 训练目标](./02-objective.mdx) | 还是「预测下一个词」：答案藏在文本自身；把预测做好，就被迫学会语法、事实和推理。 |
| [4.3 Scaling Laws](./03-scaling-laws.mdx) | C ≈ 6ND，损失按幂律可预测下降；Chinchilla 最优约 20 token/参数，2024 后主流故意「过训练」换推理省钱。 |
| [4.4 并行训练](./04-parallelism.mdx) | 显存大头是优化器状态（12/16 字节）；DP 提速不省显存，TP 凶通信限机内，PP 有气泡，ZeRO 切三大件。 |
| [4.5 训练稳定性](./05-stability.mdx) | bf16 干活 + fp32 记账；梯度裁剪限速；loss spike 就回滚跳过；高频 checkpoint 是生命线。 |

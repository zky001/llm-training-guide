# 6.2 KV Cache：不重算昨天的作业

## 第 1000 个字的荒唐账

想象模型正在生成一篇长回答，写到第 1000 个字。按 [3.2 节](../03-transformer/02-qkv.mdx)的注意力机制，这个新字要和**前面全部 999 个字**打交道：拿自己的 Q，去匹配它们每一个的 K，再加权取用它们的 V。

问题来了：那 999 个字的 K 和 V 向量，跟生成第 999 个字时算的**一模一样**——token 没变、权重没变，算出来的 K、V 就不会变。如果每写一个新字都把全部历史的 K、V 重算一遍，就等于每天上学前把从小学一年级到昨天的作业全部重写一遍。

傻子才重算。把算过的 K、V 存进显存，新字来了直接取用——这就是 <Term id="kv-cache">KV Cache</Term>，所有大模型推理服务的第一块基石。

## 天下没有免费的显存

省下的计算不是凭空来的——代价是**显存**。实验里右边面板的「缓存占用」一行行往上加：每多一个 token，就多存一行 K、V。

这笔账在真实规模下相当吓人：70B 级模型如果每个注意力头都存自己的 K、V（早期的设计），聊到 3 万 token 时，光缓存就要**几十 GB**——比半个模型还大。现代模型靠「多头共享 KV」（细节在深入层）把它砍到十分之一，但 KV Cache 依然是长对话场景的显存大头。

这也直接回答了一个日常疑问：**为什么长上下文那么贵？** 上下文越长，缓存越大、每步注意力要扫的历史越多——显存和算力双重涨价（8.2 节还会回到这个话题）。

:::caution 这个演示在哪里简化了？
实验只画了**一层、一个头**的注意力，而且只有 10 个 token；真实模型每一层的每个 KV 头都有自己的缓存，总量是「层数 × 头数」倍。另外，「无缓存」的实现现实中没人真的用——它存在的意义是让你看清缓存到底省掉了什么。最后，格子数对应注意力计算次数的相对比较，不是精确的 FLOPs。
:::

<DeepDive title="KV Cache 的显存公式">

**缓存大小**（字节）：

$$
\text{Cache} = 2 \times L \times H_{kv} \times d_{head} \times n \times B \times \text{bytes}
$$

逐项拆解：$2$ 是 K 和 V 各一份；$L$ 是层数（每层都要缓存）；$H_{kv}$ 是 **KV 头数**；$d_{head}$ 是每头维度；$n$ 是序列长度；$B$ 是批大小；bytes 是精度（bf16 为 2）。

**代入算两笔（量级示意）：**

- **GPT-2 小模型**（2019）：$L=12$，$H_{kv}=12$，$d_{head}=64$，$n=1024$，$B=1$，bf16：$2 \times 12 \times 12 \times 64 \times 1024 \times 2 \approx 38$ MB——毛毛雨。
- **70B 级、原始多头注意力**（MHA，每个头都存）：$L=80$，$H_{kv}=64$，$d_{head}=128$，$n=30000$：约 **79 GB**——比模型权重的一半还多，一张卡直接被缓存挤爆。

**为什么只缓存 K、V，不缓存 Q？** 注意力里历史 token 的 K、V 会被**之后每一个**新 token 反复查询，而每个 token 的 Q 只在它自己生成的那一步用一次——用完即弃，没有缓存价值。

**GQA / MQA：把 $H_{kv}$ 砍下来。** 分组查询注意力（GQA，2023）让多个 Q 头共享一组 K、V 头——比如 64 个 Q 头共享 8 个 KV 头，缓存立减 8 倍：上面 79 GB 的账立刻变成约 10 GB。极端版 MQA 只留 1 个 KV 头。代价是轻微的质量损失，收益是缓存和带宽的成倍节省——这就是 [3.5 节](../03-transformer/05-gpt-assembly.mdx)提过的「推理友好」架构改动，2023 年后新模型的标配。

</DeepDive>

## 小结

:::tip 本节要点
- 历史 token 的 K、V 永远不变——KV Cache 把它们存起来，每步只算新 token 的一行。
- 计算量从每步「整个三角形」塌缩成「一行」，长文本下节省成百上千倍。
- 代价是显存随上下文**线性**膨胀，长对话里缓存常比模型权重还占地方——GQA 等共享技术就是为砍它而生。
- 「长上下文为什么贵」的直接原因就在这里。
:::

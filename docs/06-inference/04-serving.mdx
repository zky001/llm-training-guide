# 6.4 推理服务框架：一台服务器伺候一千个人

## 同一张卡，差出几十倍

做个实验：用 Python 的 transformers 库写十行代码，把一个 7B 模型跑起来对外服务；再用 vLLM 这样的专业推理框架跑同一个模型、同一张卡。压测结果常常让人怀疑硬件被调包了——**每秒能服务的用户数，能差出几倍到几十倍**。

模型没变、显卡没变，差距全在工程。这一节讲专业框架的两大魔法，它们分别堵住了朴素服务的两个大漏洞。

## 漏洞一：先写完的乘客在干等

GPU 喜欢成批干活（第 4 章讲过：批越大，算力利用率越高），所以服务器会把多个用户的请求拼成一批同时生成。但朴素的做法是「整批进、整批出」——就像一辆**旅游大巴**：等 32 位乘客全部上车才发车，开到终点才统一放人。

问题是回答有长有短：有人的回答 10 个 token 就写完了，有人要写 1000 个。先写完的只能坐在座位上干等，那个位置的算力全部白烧；门外排队的新请求，也必须等整车到站才能上车。

**<Term id="continuous-batching">连续批处理</Term>**把大巴换成了**地铁**：调度以「生成一步」为单位——每往前走一步，谁的回答结束了就立刻下车，空出的位置马上让排队的新请求上来。GPU 上永远坐满干活的乘客，没有一个空座。

## 漏洞二：显存被「包间制」浪费

第 [6.2 节](./02-kv-cache.mdx)说过，<Term id="kv-cache">KV 缓存</Term>是推理显存的大户。朴素实现还要雪上加霜：服务器不知道每个用户会聊多长，只好给每个请求**预留一整块「最大可能长度」的连续显存**——像酒店不管来一个人还是十个人，一律开最大的宴会厅。

结果就是：大部分预留空间空着；而且长短不一的「包间」进进退退，显存被切得七零八落——明明总量还够，却找不出一块连续的大空间给新请求（碎片化）。vLLM 团队实测（2023 年论文数据）：传统系统里真正装着有用数据的 KV 显存**只有两三成**，其余全在陪跑。

**<Term id="paged-attention">分页注意力</Term>**（PagedAttention，vLLM 2023）把操作系统课搬了过来：别开包间了，把每个请求的缓存切成固定大小的**小页**，用到哪页才分配哪页；零散的物理页靠一张「页表」串成逻辑上连续的序列。浪费从 60% 以上降到 4% 以下（论文数字）。

显存省下来 = 同一张卡能同时伺候更多请求 = <Term id="throughput">吞吐量</Term>成倍上涨。两招合力，vLLM 论文报告的吞吐相对当时的常规服务方式**提升了一个数量级以上**（2023 年数字，具体倍数随场景变化）。

如今这套思想已是行业标配：vLLM、SGLang、TensorRT-LLM 等主流推理框架（2025 年生态）各有绝活，但地基都是这两块。更进一步，由于<Term id="prefill">预填充</Term>吃算力、<Term id="decode">解码</Term>吃带宽（[6.1 节](./01-train-vs-infer.mdx)的结论），2024-2025 年不少大厂干脆把两个阶段**拆到不同的机器**上各自优化——「prefill/decode 分离」。

:::caution 这些数字的保质期很短
本节引用的浪费比例、加速倍数都来自特定时间点的论文与实测（已标注年份），框架每个版本都在进步，具体数字请以官方文档为准。另外，真实推理系统的工程远不止这两招——量化内核、投机解码、请求优先级、抢占与降级……本节只讲清「为什么快」的两个核心思想。
:::

<DeepDive title="PagedAttention 与调度细节">

**块表：KV 缓存的「页表」。** PagedAttention 与操作系统虚拟内存的对应关系几乎是一比一的：

| 操作系统 | PagedAttention |
| --- | --- |
| 虚拟地址空间 | 请求的逻辑 KV 序列 |
| 物理内存页（page） | 显存里的 KV 块（block，如每块 16 个 token） |
| 页表 | 块表（block table）：逻辑块号 → 物理块号 |
| 按需分页（demand paging） | 生成到哪个 token 才分配哪个块 |

注意力计算时按块表把零散的物理块「串」回连续序列，数学结果与连续存储完全相同——改变的只是内存管理，不是算法。

**前缀共享（prefix sharing）。** 很多请求的开头是相同的——最典型的是同一个 system prompt。分页之后，这些相同前缀的 KV 块可以**多个请求共享同一份物理页**（配合写时复制 copy-on-write），几千个并发用户的 system prompt 在显存里只存一份。few-shot 示例、多轮对话的公共历史同理。

**连续批处理的调度粒度。** 关键设计是 **per-iteration scheduling**：每生成一步都重新决定「这一步的批里有谁」——完成的请求退出、新请求插入、必要时抢占低优先级请求（把它的 KV 页换出到内存）。对比传统的 per-request scheduling（批的成员从头到尾固定），GPU 空转时间从「批内最长回答决定」降到几乎为零。

**投机解码是另一条战线。** 上面两招提高的是「多用户总吞吐」；对单个用户的生成速度（decode 受显存带宽限制，见 [6.1 节](./01-train-vs-infer.mdx)深入层），主流方案是投机解码：小模型快速打草稿、大模型一次并行验证多个 token——把「一步搬一次权重」摊薄成「一步验证多个词」。

</DeepDive>

## 小结

:::tip 本节要点
- 朴素服务的两大浪费：**先完成的请求占着座位干等**（静态批处理）、**KV 显存包间制**（预留+碎片化）。
- 连续批处理按「生成步」调度，随下随上，GPU 永远满载。
- PagedAttention 用操作系统分页的思路管理 KV 缓存，显存浪费从 60%+ 降到 4% 以下（2023 年论文数字）。
- 显存省下来就能塞进更多并发请求——省显存与提吞吐是同一件事。
:::

# 5.4 PPO：一场小心翼翼的拔河

## 一个看似简单的计划

上一节结束时，我们手里有了一个会打分的<Term id="reward-model">奖励模型</Term>。接下来的计划看起来简单得不像话：

> 让模型可劲儿生成回答，奖励模型打分，往高分方向调参数——完事。

方向也熟悉：这不就是 [0.4 节](../00-prerequisites/04-gradient.mdx)的梯度下降换了个方向吗？损失要下山，奖励要**上山**，一步步爬就是了。

真按这个计划跑三个月，你大概率会得到一个怪物：满嘴「当然可以！这真是一个非常棒的问题！！」的马屁精，每个回答不灌到八百字不罢休，通篇正确的废话，自信得像什么都懂，其实什么都没说。

## 学生没变好，它只是摸透了裁判

出问题的不是优化算法——优化算法工作得**太好了**。问题出在裁判身上。

奖励模型是从人类偏好里学出来的，而人类的偏好带着系统性的小毛病。你在[上一节的标注实验](./03-rlhf.mdx)里亲手见过两个：第 2 组里，更长、信息更多的回答看起来更「丰富」（哪怕混进了编造）；第 5 组里，顺着用户说的回答更「讨喜」。几十万条标注平均下来，这些偏差被奖励模型忠实地继承，再被强化学习**疯狂放大**：

- 裁判略偏爱长回答 → 学生把每个回答灌水到极限；
- 裁判略偏爱自信语气 → 学生戒掉了「可能」「我不确定」；
- 裁判略偏爱附和 → 学生变成职业马屁精。

这个现象有个名字：<Term id="reward-hacking">奖励钻空子</Term>。记住它的形状：**只要评分标准有漏洞，一个足够强的优化器就一定会找到并利用它**——刷的是分数，不是能力。这不是大模型特有的毛病，考试应试、KPI 造假，都是同一个故事。

## 解法：在拔河的另一头拴根橡皮筋

既然放任优化会跑偏，那就别让它跑远。办法朴素得可爱：给模型拴一根橡皮筋，另一头系在**原来的它**（SFT 模型）身上。

模型每偏离原来的说话方式一分，就从奖励里扣一分。这根橡皮筋叫 <Term id="kl-penalty">KL 惩罚</Term>，整场拔河的目标变成：

$$
\text{总目标} = \text{奖励} - \beta \times \text{偏离程度}
$$

$\beta$ 是橡皮筋的松紧：拉太紧，模型不敢改进；放太松，马屁精卷土重来。于是优化的含义变成——**在「别变得面目全非」的前提下，尽量讨裁判喜欢**。灌水八百字、句句感叹号的文风离 SFT 模型太远，橡皮筋会把它拽回来。

## PPO：连每一步都要小心

橡皮筋管住了「总体别跑偏」，<Term id="ppo">PPO</Term> 则把「每一步别迈太大」也做进了算法里。它的全名是「近端策略优化」——「近端」二字直译过来就是**别走远**：每次参数更新都被夹在一个安全范围内，涨得太猛就截断（很有 0.4 节「学习率太大会飞出山谷」的既视感）。

代价是工程上的沉重。一次 PPO 训练要在显存里同时伺候**四个模型**：被训练的模型、用来算橡皮筋的参考模型、打分的奖励模型，还有一个估计「这个回答比平均水平好多少」的价值模型。又贵、又娇气、超参数难调——RLHF 因此成了大厂的特权游戏。这份沉重，正是下一节各种「捷径」诞生的理由。

:::caution 这一节的边界在哪里？
本节给的是 PPO 的直觉骨架，不是全貌：强化学习的细节（优势估计、采样效率、训练稳定性技巧）水很深，跳过了整整一门课的内容。另外「四个模型」是经典配置，不同实现有变体——有的把奖励模型和价值模型合并，有的用冻结技巧省显存。方向对了就好，细节请以论文和代码为准。
:::

<DeepDive title="RLHF 目标函数与 PPO 机制">

**总目标的正式写法。** RLHF 阶段的优化目标是：

$$
\max_{\pi_\theta}\; \mathbb{E}_{x \sim \mathcal{D},\; y \sim \pi_\theta(\cdot|x)}\bigl[\, r_\phi(x, y) \,\bigr] \;-\; \beta\, \mathbb{D}_{\mathrm{KL}}\!\bigl(\pi_\theta(\cdot \mid x) \,\big\|\, \pi_{\mathrm{ref}}(\cdot \mid x)\bigr)
$$

逐项翻译：$\pi_\theta$ 是正在被训练的模型（策略），$x$ 是问题、$y$ 是模型自己采样出的回答；$r_\phi$ 是奖励模型的打分；$\pi_{\mathrm{ref}}$ 是冻结不动的参考模型（通常就是 SFT 模型）。KL 散度 $\mathbb{D}_{\mathrm{KL}}$ 可以粗糙地理解为**两个概率分布之间的「距离」**：新模型的下一词分布和参考模型差得越多，它越大。$\beta$ 就是正文里的橡皮筋松紧。

**PPO 的「别走远」怎么实现。** PPO 每轮先用旧参数采样一批回答，再做多步小更新。核心是重要性采样比率：

$$
r_t(\theta) = \frac{\pi_\theta(a_t \mid s_t)}{\pi_{\theta_{\mathrm{old}}}(a_t \mid s_t)}
$$

它衡量「新模型对这个 token 的偏好比旧模型变了多少倍」。PPO 的目标函数把它夹住：

$$
L^{\mathrm{CLIP}}(\theta) = \mathbb{E}_t\Bigl[\min\bigl(r_t(\theta)\,\hat{A}_t,\;\; \mathrm{clip}\bigl(r_t(\theta),\, 1-\epsilon,\, 1+\epsilon\bigr)\,\hat{A}_t\bigr)\Bigr]
$$

人话版：$\hat{A}_t$ 是「优势」——这个动作比平均水平好多少（好为正、差为负）。想沿着优势方向更新可以，但比率 $r_t$ 一旦超出 $[1-\epsilon,\, 1+\epsilon]$（$\epsilon$ 通常取 0.2），就按边界截断，多余的梯度直接作废——**再大的诱惑也只许迈一小步**。

**价值模型是干嘛的。** 优势 $\hat{A}_t$ 需要一个基准：「平均水平」是多少。价值模型（通常和被训模型同尺寸）就负责估计这个基准。它是四模型账单里最容易被嫌弃的一个——下一节的 GRPO 干脆把它裁掉了。

**四模型显存账。** 以 70B 模型做 RLHF 为例：被训模型（参数+梯度+优化器状态）+ 参考模型（只推理）+ 奖励模型（只推理）+ 价值模型（参数+梯度+优化器状态），粗算下来比同尺寸 SFT 贵好几倍，还要协调「生成—打分—更新」的流水线。这就是「RLHF 比 SFT 贵一个量级」的由来。

**Reward hacking 不是假想敌。** InstructGPT 论文（2022）就报告了过度优化奖励模型会损害真实质量的现象，需要用 KL 惩罚和早停控制；后续研究（如 2023 年的 reward model overoptimization 定量研究）进一步表明：奖励模型分数一路上涨时，人类真实评价会先升后降——裁判被刷爆了。

</DeepDive>

## 小结

:::tip 本节要点
- 只追奖励必出事：评分标准的任何漏洞都会被优化器找到并放大（reward hacking）。
- KL 惩罚是拴向 SFT 模型的橡皮筋：在「别变得面目全非」的前提下讨裁判喜欢。
- PPO 的「近端」= 每步更新都被夹在安全范围内，稳但工程沉重（四个模型同时在显存）。
- 裁判的偏差来自人类标注的偏差——你在 5.3 实验里亲手见过它们的源头。
:::

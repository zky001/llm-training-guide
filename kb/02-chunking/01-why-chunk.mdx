# K2.1 为什么要切块，又怎么切坏

## 为什么不能整篇塞进去

你有一份 50 页的员工手册，要放进知识库。最偷懒的想法是：整篇存进去、整篇检索。为什么不行？两个硬约束把这条路堵死了。

**约束一：检索要精准。** 用户问「年假有几天」，你该给模型的是「年假」那一小段，而不是把整本手册甩过去。为什么？因为把 50 页全塞进模型的上下文，既贵（上篇 [8.2 节](/docs/frontier/long-context)算过，长上下文又慢又费钱），又蠢——年假那一句被埋在几万字里，重点被大量无关内容稀释，模型反而容易「看走眼」。给它精准的一段，它才答得准。

**约束二：向量装不下整篇。** 回想 [K1](../01-semantic-search/index.md)：语义检索靠的是「把资料变成一个向量」。可一篇 50 页的长文，压成**一个**向量会怎样？细节全糊了。上篇 [1.4 节](/docs/language-models/embedding)讲过，一个向量的表达能力是有限的——它能概括「这篇大致讲人事制度」，却记不住「年假 5 天、病假需医院证明、婚假 3 天」这些具体事实。用一个模糊的向量去检索，什么都对得上一点，什么都不精准。

所以，长文档必须先<Term id="chunking">切块</Term>：切成一段段小的<Term id="chunk">文档块</Term>，每块单独编码成向量、单独入库。检索时,匹配到哪块就取哪块。

## 两个极端，两种坑

切块的核心矛盾，藏在「块该多大」这个问题里。往两个极端走，各有各的死法：

- **切太大**：一块里塞了太多不同的事。检索时就算命中了，捞回来的也不干净——你要的那句话，混在一堆无关内容里；而且一块内容太杂，它的向量会被「平均」得谁都不像，相似度反而拉低。
- **切太小**：一个完整的事实，被块的边界拦腰截断。比如「会员 15 天无理由退货」在一句、「但生鲜、定制不能退」在另一句——切开之后，检索可能只捞回前半句，模型就会自信地告诉你「都能退」，把关键的例外漏得干干净净。

## 从实验里读出三条教训

**先玩第一个查询「会员退货，有什么东西是不能退的？」**，点「开始检索」，三种切法三种结局：

- **① 整篇一大块**：捞回了整篇——答案确实在里面，但混着运费、退货操作等一堆无关内容（🌀 噪声多）。文档一大，重点就被稀释、相似度被拉平。
- **② 每句一小块**：💥 只命中了「15 天无理由」那句（它和「会员退货」字面最像），却把关键的「例外清单」那句漏在了另一个块里。结果模型会告诉你「会员都能退」——把生鲜、定制不能退这个要命的例外，漏得一干二净。这就是**切太碎，切断了完整语义**。
- **③ 按语义切**：把「时长 + 例外」这两句放进了同一块，所以答案完整：会员 15 天无理由，但生鲜、定制、拆封数码除外（✅）。

**再玩第二个查询「退货要自己付运费吗？」**，你会发现一个反转：这次「② 每句一小块」也答对了！因为「运费」这个事实，本来就完整地待在一句话里，没有跨越块的边界。

这个反转很重要，它揭示了切块真正的原则：

:::info 好切块的唯一标准
**让每一块 = 一个完整的意思。** 既不掺太多别的事（否则重点被稀释），也不把一件事切散（否则只捞回半个答案）。「切太碎」本身不是错——只有当它把一个完整的事实切断时，才致命。
:::

## 切块的两个基本旋钮

实践中，最常见的切块方式就是「按固定长度切」，它有两个要调的旋钮：

- **块大小（chunk size）**：一块多长，通常按 token 数算（比如 256~512 token）。这是最关键的超参数。顺带记个换算：中文里 1 个汉字大致对应 1~2 个 token（取决于[分词器](/docs/language-models/tokenization)），所以 512 token 差不多是 300~500 个汉字——你可以拿自己文档的字数除一下，估出大概能切多少块。
- **重叠（overlap）**：相邻两块之间留一段重叠内容（比如后一块的开头，重复前一块的结尾 50 个字）。这样即使某个句子正好压在边界上，它至少在某一块里是完整的——像给切口打了个「补丁」。

:::caution 切块没有「万能最优解」
不要指望有一个「切块大小 = 400」的标准答案。最优切法高度依赖你的**文档类型**（手册按小节切、对话按轮次切、代码按函数切、财报按表格切）和**查询类型**（问细节要小块，问概括要大块）。chunk size 是一个必须**拿真实查询去测、去调**的超参数，不是拍脑袋定的——这也是为什么业界常把切块称作 RAG 里最需要「手艺」和耐心的一步。垃圾的切块，会让后面所有环节都白费力气。
:::

<DeepDive title="切块策略与 chunk size 的权衡">

**四类常见切块策略**（由粗到精）：

1. **固定大小切块（fixed-size）**：按字符数或 token 数硬切，配一段 overlap。最简单、最常用、最省事——很多生产系统的起点就是它。缺点是完全不管语义，专在句子中间下刀。
2. **按分隔符切（recursive/delimiter）**：优先在自然断点切（先按段落，段落太长再按句子，句子太长再按标点），尽量不破坏句子完整性。
3. **按结构切（structure-aware）**：尊重文档的天然边界——Markdown 按标题层级、HTML 按标签、代码按函数/类。结构清晰的文档用它效果最好。
4. **按语义切（semantic）**：用嵌入相似度探测「话题在哪里转折」，在转折处下刀（下一节 [K2.2](./02-smart-chunking.mdx) 细讲）。

**chunk size 的三方权衡**——它没有免费的午餐：

| 方向 | 好处 | 代价 |
| --- | --- | --- |
| 块偏大 | 上下文完整、跨句事实不易断 | 检索精度降、噪声多、生成成本高 |
| 块偏小 | 检索精准、命中干净、成本低 | 上下文碎、跨块事实被切断 |

**overlap 的作用与代价**：重叠能显著减少「边界截断」——边界处的句子在相邻块里各有一份完整拷贝，检索到哪块都不缺。代价是存储和索引冗余（同一段内容被存了两遍），以及检索时可能返回内容高度重复的相邻块。典型 overlap 取块大小的 10%~20%。

**别忘了元数据。** 每个块入库时,除了向量,还该附上元数据:它来自哪篇文档、哪个章节、什么日期。这为两件事铺路——[K5](../05-generation/index.md) 的引用标注（答案要能指回原文出处），和 [K3](../03-retrieval-methods/index.md) 的带过滤检索（「只在 2025 年的文档里找」）。

**最后，评测驱动调参。** chunk size、overlap、切法，都不该拍脑袋。正确姿势是准备一批真实的「问题→期望答案」对，用不同配置各跑一遍，用检索命中率和端到端答对率来选——这正是 [K6 评测](../06-evaluation/index.md)要教的方法。切块调参是 RAG 优化里投入产出比最高的环节之一。

**最朴素的固定大小切块，就这几行**（带 overlap 的字符级滑窗）：

```python
def chunk(text, size=400, overlap=80):
    """按字符切，相邻块重叠 overlap 个字符，防止句子被边界切断。"""
    chunks, start = [], 0
    while start < len(text):
        chunks.append(text[start : start + size])
        start += size - overlap        # 下一块的起点往回退 overlap，制造重叠
    return chunks

# 生产里通常按 token 数切（更贴合模型），并在自然断点（段落/句号）优先下刀，
# 但内核就是这个「滑窗 + 重叠」。
```

## 小结

:::tip 本节要点
- 长文档必须切块：整篇检索既稀释重点、又浪费上下文，且一个向量装不下整篇的细节。
- 切太大→命中也不干净、相似度被拉平；切太小→完整的事实被边界切断、只捞回半个答案。
- 好切块的唯一标准：**每一块是一个完整的意思**。切太碎不是永远错，只有切断完整事实时才致命。
- chunk size 和 overlap 是要**拿真实查询调**的超参数；切块没有万能最优解，高度依赖文档和查询类型。
:::

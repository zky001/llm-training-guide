# K7.2 Agentic RAG：让模型自己决定查几轮

## 「查一次」不够用的时候

到目前为止，我们的 RAG 都是「一根筋」：检索一次、拼进去、答完收工。这对「一个问题、一段答案」的场景很好用。可真实问题常常没那么听话：

- 「**对比一下** A 和 B 两款产品的退货政策」——得查两次：A 的政策一次，B 的政策一次，才能对比；
- 「**去年 Q3 的营收**是多少」——你可能得先查到「财报在哪个文档、哪个季度」，再去查具体那个数字，两步走；
- 检索回来一看**资料不够**——那本该改写查询、再查一轮，而不是拿着残缺资料硬答。

固定的「查一次」，应付不了这些需要**多步、对比、迭代**的问题。

## 把检索从「一步」升级成「一个工具」

<Term id="agentic-rag">Agentic RAG（智能体式 RAG）</Term>的思路很直接：**别再把检索当成固定的前置一步，而是当成模型可以自主使用的工具。** 模型自己决定——要不要查？查什么？查回来够不够？要不要换个说法再查一轮？

看出来了吗——**这正是下篇智能体的思路**。检索是一个工具，模型在「思考 → 行动 → 观察」的循环里反复调用它。事实上，这就是下篇 [A3.3 Agentic RAG](/agents/memory-context/agentic-rag) 讲的东西：

> **中篇的检索技术（K1~K6）+ 下篇的智能体循环 = Agentic RAG。** 三部曲在这里交汇了。

Agentic RAG 由此获得了朴素 RAG 没有的几种能力：

- **查询分解**：把「对比 A 和 B」拆成两个子查询，分别检索，再综合；
- **迭代检索**：查完发现不够、或不相关，就改写查询、再查一轮，直到够了为止；
- **自我核对**：把多次检索的结果交叉验证，发现矛盾时再深挖。

**代价也很实在。** 更强意味着更贵更慢：多轮检索 = 多轮模型调用 + 多轮检索开销（回扣下篇 [A7 的成本账](/agents/agent-evaluation/error-compounding)）。而且多轮还意味着**错误会累积**——第一轮查偏了，后面越走越歪（下篇 A7 讲的「错误复利」在这里同样成立）。

:::caution 别把 Agentic RAG 当成无脑升级
简单的单跳问答——「退货政策是什么」——用朴素 RAG 又快又好，套上一整个智能体循环纯属浪费。什么时候才值得上 Agentic RAG？**问题需要多步、需要对比、需要「先找线索再找答案」的时候。** 判断口诀和下篇 [A0.2](/agents/from-chat-to-action/autonomy-spectrum) 一模一样：固定流程能搞定的，就别上智能体；只有当「查几轮、查什么」没法预先写死、得让模型看情况现定时，Agentic RAG 才划算（2025-2026 的普遍经验）。
:::

<DeepDive title="从朴素 RAG 到 Agentic RAG 的谱系">

RAG 这些年沿着「让检索越来越智能」的方向演进，是一条清晰的谱系：

1. **朴素 RAG**：检索一次 → 拼接 → 生成。固定、简单，单跳问答够用。
2. **带查询改写/分解的 RAG**：先让模型把用户的口语问题改写成更好检索的形式，或拆成子查询——检索还是「前置步骤」，但更聪明了。
3. **迭代式 RAG**（self-RAG、CRAG 等，2023-2024）：模型**评估检索质量**——「这批资料够不够、相不相关？」——不够就重新检索。检索开始有了「反馈闭环」。
4. **完全 Agentic RAG**：检索彻底变成一个工具，模型在循环里**自主编排**——什么时候查、查几次、怎么综合，全由它临场决定（回扣下篇 [A0 行动循环](/agents/from-chat-to-action/what-is-agent)、[A1 工具调用](/agents/tool-use/function-calling)）。

**self-reflection 检索**是其中的关键一环：检索回来后，模型先自问「这够回答问题吗？相关吗？」，再决定下一步是「够了，作答」还是「不够，再查」。这把 [K6 的评测思路](../06-evaluation/index.md)（三元组里的上下文相关性）内化进了运行时。

**和下篇多智能体的关系**：更复杂的系统里，检索本身可以是一个专门的「检索智能体」，和「写作智能体」协作——一个负责把资料找全找准，一个负责综合成答案（回扣下篇 [A5 多智能体](/agents/multi-agent/why-multi)）。

一句话：Agentic RAG 不是一个具体技术，而是「让模型对检索有掌控权」的一整个方向——它把中篇的「怎么检索得准」和下篇的「怎么让模型自主行动」缝在了一起。

</DeepDive>

## 小结

:::tip 本节要点
- 固定的「查一次」应付不了需要**多步、对比、迭代**的问题。
- **Agentic RAG** 把检索从「前置一步」升级成「模型自主使用的工具」，让模型自己决定查什么、查几轮、够不够。
- 它就是**中篇检索 + 下篇智能体循环**的结合（接下篇 [A3.3](/agents/memory-context/agentic-rag)），三部曲在此交汇。
- 代价是更贵更慢、错误会累积；单跳简单问答别无脑上——固定流程能搞定就别上智能体。
:::

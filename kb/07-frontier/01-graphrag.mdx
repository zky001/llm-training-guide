# K7.1 GraphRAG：回答「拼不出来」的问题

## 一个普通 RAG 答不出的问题

到现在为止，我们的 RAG 都在做同一件事：拿问题去找**和它最像的那几段资料**（[K1](../01-semantic-search/index.md)）。这对「答案就写在某一段里」的问题很好用。可有些问题偏偏不是这样：

> 「张伟的经理的助理是谁？」

翻遍知识库，**没有任何一段**直接写着这个答案。它藏在三段资料的关系链里：张伟属于技术部、技术部的经理是李娜、李娜的助理是王芳。你得把这三段**串起来**才能推出「王芳」。这叫<Term id="multi-hop">多跳推理</Term>——A→B、B→C，所以 A→C。

而单段相似检索，恰恰跨不过这个坎。亲眼看一遍：

## 为什么普通 RAG 会卡壳

玩过实验你会发现：普通 RAG 模式下，检索按「和问题的相似度」排序，可问题「张伟的经理的助理是谁」里**压根没提「技术部」「李娜」**——于是中间那两跳「技术部的经理是李娜」「李娜的助理是王芳」相似度太低、根本捞不回来。它只拿到了链条的头（张伟属于技术部），然后就卡住了。

**根子在于**：相似检索找的是「和问题像的段」，可多跳问题的中间环节，字面上和问题一点都不像。这是朴素 RAG 的结构性软肋。

## GraphRAG 的思路：先连成图，再顺着走

<Term id="graphrag">GraphRAG</Term> 换了个存法。它不满足于把资料切成一段段孤立的块，而是先从文档里把**实体和关系**抽出来，连成一张知识图谱：

```
张伟 —属于→ 技术部 —经理→ 李娜 —助理→ 王芳
```

检索时，它不只找相似段，还能**顺着关系边一跳跳走**。切到实验的 GraphRAG 模式，点「顺着关系走一遍」——它从「张伟」出发，沿「属于→经理→助理」三跳到位，稳稳走到「王芳」。多跳问题，迎刃而解。微软在 2024 年把这套思路做成了开源方案 GraphRAG，带火了这个方向。

**但它有代价。** 建图谱要先花一大笔力气：用大模型从每篇文档里抽实体、抽关系、去重、连边——又慢又贵。所以 GraphRAG 不是来取代普通 RAG 的，而是**专治「关系密集、多跳」的硬骨头**：梳理整个组织架构、跨多份文档追一条因果链、回答「把这批文档整体串起来看，讲了什么」这类全局问题。

:::caution 别为了炫技上 GraphRAG
两个提醒。其一：图谱质量取决于抽取质量——要是大模型把关系抽错了（把「李娜的助理是王芳」抽成「王芳的助理是李娜」），顺着走只会走进沟里，这是[垃圾进垃圾出](../00-why-rag/index.md)的新版本。其二：现实中**很多日常问答是单跳的**（「退货政策是什么」），普通 RAG 又快又好，硬上 GraphRAG 纯属杀鸡用牛刀。2024-2026 年 RAG vs GraphRAG 的评测结论也是这样：各有胜场，看问题类型——多跳、全局、关系型问题 GraphRAG 强，简单事实问答普通 RAG 更划算。
:::

<DeepDive title="知识图谱的构建与社区检测">

**图谱是怎么建出来的？** 核心是**实体关系抽取**：让大模型逐段读文档，抽出「主-谓-宾」三元组（如 `(技术部, 经理, 李娜)`）。这一步的两个难点：

- **实体去重**：同一个实体在不同文档里叫法不同（「张伟」「张经理」「小张」可能是同一人），得合并成同一个节点，否则图是断的。
- **关系消歧**：抽取容易出错，需要校验和置信度过滤。

**微软 GraphRAG 的招牌是「社区检测 + 分层摘要」**（2024）：把建好的图谱用社区检测算法聚成一个个「社区」（联系紧密的实体簇），再给每层社区生成摘要。这样它不仅能答具体的多跳问题，还能答**全局性问题**——「这批文档整体在讲什么」「有哪几个主要主题」——这恰恰是普通 RAG 最不擅长的（它只会检索局部片段，没有全局视野）。

**图检索常和向量检索混用**：先用向量检索定位「入口节点」（问题相关的起点实体），再从入口开始图遍历。这样兼顾了向量的「模糊入口」和图的「精确多跳」。

**适用边界**：GraphRAG 建图成本高、维护复杂，适合关系密集、需要多跳或全局理解的知识库（组织架构、供应链、论文引用网络、案情梳理）；对以「独立事实查询」为主的场景，性价比不如朴素 RAG + [混合检索](../03-retrieval-methods/index.md) + [重排](../04-reranking/index.md)。

</DeepDive>

## 小结

:::tip 本节要点
- **多跳问题**：答案没写在任何单段里，得串起好几段才能推出来（张伟→技术部→李娜→王芳）。
- 普通 RAG 卡壳，是因为中间环节字面上和问题不像、相似检索捞不回来。
- **GraphRAG** 先把资料抽成「实体+关系」图谱，检索时顺着关系边遍历，多跳迎刃而解（微软 2024 开源）。
- 代价是建图又慢又贵；它专治关系密集、多跳、全局型问题，别拿它做简单事实问答。
:::

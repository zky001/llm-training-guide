# K7.3 RAG vs 长上下文：2026 年的真相

## 一个流传甚广的预言

上下文窗口从 2020 年的几千 token，一路卷到 2026 年的百万级（上篇 [8.2 节](/docs/frontier/long-context)）。于是有人放话：

> 既然模型一次能塞进几十万字，何必费劲搞什么检索、切块、重排？把整个知识库一股脑塞给它，让它自己在里面找不就完了——**「长上下文会杀死 RAG」**。

听起来很有道理。是真的吗？拖动滑块算笔账，你就明白了：

## 账一算，预言就破产了

玩过计算器，三件事一目了然：

**其一，成本随知识库线性爆炸。** 把「知识库大小」滑块往大拖，「长上下文全塞」的成本直冲云霄——因为它**每一次查询**都要为整个知识库付费；而 RAG 每次只塞检索到的那几段，成本几乎不随库变大。库越大，差距越夸张（几十倍、几百倍）。

**其二，总会撞上窗口上限。** 库超过约百万 token（2026 旗舰模型的窗口量级），「全塞进去」直接**塞不下**了——计算器会亮「超窗口」。知识库只会越来越大，这条路迟早走死。

**其三，读越多越走神。** 就算塞得下、付得起，上下文越长，模型越容易「中间迷失」——埋在中间的关键信息被忽略（回扣上篇 [8.2 的「标称≠有效」](/docs/frontier/long-context)和 [K5.1 的 lost-in-the-middle](../05-generation/01-context-assembly.mdx)）。塞得多，不等于用得好。

**再加上第四点**：知识库要**实时更新**（改一份文档就生效），总不能每次查询都把整个最新的库重塞一遍。

不过，计算器也诚实地告诉你一件事：把库拖**小**，会看到「小库是长上下文的甜区」——当知识库小到能轻松塞下时，直接全塞反而更省事（不用维护检索、也不怕漏召回）。**RAG 的价值，随知识库变大才凸显。**

## 2026 的共识：不是替代，是搭档

所以「长上下文杀死 RAG」的预言破产了。取而代之的 2026 共识是——**两者混用**：

> **用检索决定「该看哪些资料」（把知识库缩小到相关的一小撮），再用长上下文「在这一小撮上好好推理」。**

RAG 从「长上下文的对手」，变成了「长上下文的**最佳搭档**」：检索负责在海量知识里精准定位，长上下文负责在定位出的证据集上深度理解。计算器里的「混合」方案，正是这个思路——它既避开了全塞的天价成本和窗口上限，又拿到了长上下文「一次读一大片、能综合推理」的好处。

:::caution 这条边界在移动
RAG 和长上下文的权衡，随着窗口变大、token 变便宜，**一直在变**——今天的「甜区边界」明天就会移动。所以别记死「知识库超过多少 token 就该用 RAG」这种具体数字，而要记住那个判断框架：**成本、上限、精度、更新**——问自己这四个问题，答案自然浮现。这也是为什么 RAG 没有像预言的那样消失，反而活得好好的：它解决的是长上下文再大也绕不开的四个结构性问题。
:::

<DeepDive title="长上下文与 RAG 的定量权衡">

**成本的量级直觉**（回扣上篇 [4.3 的 6·N 账](/docs/pretraining/scaling-laws)的思路）：

- **长上下文全塞**：每次查询成本 ∝ 知识库大小。库有 N token，每查一次就要处理 N token，查一千次就是 1000·N。库越大越离谱。
- **RAG**：每次查询成本 ∝ 固定的检索块大小（比如几千 token），几乎与库大小无关。

**窗口是硬上限**：再大的窗口也是有限的（2026 约百万级），知识库却可以无限增长。超过窗口，「全塞」在物理上就不可能——这是 RAG 无法被替代的最硬理由。

**有效上下文 < 标称上下文**：模型宣称支持一百万 token，不代表它在一百万 token 里都「看得清」。大海捞针测试（上篇 [8.2](/docs/frontier/long-context)）反复表明，信息埋得越深、上下文越满，越容易被忽略。塞满 ≠ 用好。

**KV Cache 的账**：上下文越长，推理时的 KV Cache 显存占用线性增长（上篇 [6.2](/docs/inference/kv-cache)），长上下文的延迟和显存成本都不便宜。

**缓存复用扳回一点**：prompt caching（2024-2026）让「重复塞同样的长前缀」变便宜——如果每次查询都基于同一个大文档，缓存能省下重复处理的钱。这在一定程度上帮长上下文扳回一些性价比，但救不了「库超过窗口」和「知识要更新」这两个死结。

**混合架构的落地形态**：实践中常见的是「RAG 召回一批较大的块（比如 top-20，而非 top-3）→ 一股脑喂给长上下文模型 → 让它在这几万 token 的证据集上综合推理」。检索保证「喂进去的都相关」，长上下文保证「能一次读一大片、跨块推理」——各取所长。

</DeepDive>

## 小结

:::tip 本节要点
- 「长上下文杀死 RAG」的预言，被成本账算破产了：全塞的成本随库线性爆炸、总会超窗口、读越多越走神、还没法实时更新。
- **2026 共识是混用**：检索决定「该看哪些」、长上下文「在这一小撮上推理」——RAG 是长上下文的最佳搭档，不是对手。
- 小知识库是长上下文的甜区（能塞下就直接塞）；RAG 的价值随库变大才凸显。
- 记判断框架（成本 / 上限 / 精度 / 更新四问），别记死具体 token 数——这条边界一直在移动。
:::

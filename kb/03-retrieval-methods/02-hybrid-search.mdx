# K3.2 混合检索：两路一起上，各补盲区

## 最朴素、也最有效的主意

[上一节](./01-keyword-search.mdx)的结论是：关键词擅长精确匹配，语义擅长理解意思，两者各有各的盲区。

那最朴素的主意就是——**两个一起用，取长补短**。用户查「东西想退掉」，语义那路命中；用户查「X9-2000」，关键词那路命中；用户的问题横跨两者，就靠两路合力。这就是 2026 年生产 RAG 的标配：<Term id="hybrid-search">混合检索</Term>。

它做的事很直白：同一个查询，**关键词检索跑一遍、语义检索跑一遍**，各出一个排名，再把两个排名**融合**成最终的一个。

## 融合的拦路虎：两把尺子刻度不同

「把两个排名合起来」听着简单，真做却有个拦路虎。

关键词那路给的是 **BM25 分**（可能是 12.7、8.3 这种数），语义那路给的是**相似度**（0 到 1 之间的小数）。这两把尺子**刻度完全不同**，没法直接相加——就像一个按「分数」算、一个按「百分比」算，硬加起来毫无意义。你也不能简单归一化，因为两路的分数分布形状差很远，归一化后还是会一头重一头轻。

## RRF：干脆不看分数，只看名次

<Term id="rrf">RRF（Reciprocal Rank Fusion，倒数排名融合）</Term>用一个绝妙的偷懒办法绕开了这道坎：

> **别看分数了，只看名次。** 每条结果在各路排行榜里排第几名，名次越靠前，给的分越多（第 1 名给最多，往后递减）。把它在两个榜里的「名次分」加起来，重新排序。

为什么这招高明？因为**名次是通用货币**——不管你原来的分是 BM25 的 12.7 还是相似度的 0.83，「排第 1」就是排第 1。这样两把不同刻度的尺子，就都被翻译成了同一种语言（排名），可以直接相加了。RRF 简单到几行代码就能写完，效果却出奇地好，是工业界的默认选择（公式见深入层）。

## 上手：看混合怎么兜住单路会漏的那一半

前两个查询你已经见过——语义的主场、关键词的主场。现在看第三个，真正体现混合价值的：

<RetrievalArena />

**实验：跨话题的查询。** 选查询「生鲜的运费和退货政策」，点「三路开跑」。这条查询横跨**两个话题**：运费 + 退货。看仔细每一路——

- **语义检索**被「配送」那一带吸住了：它把运费相关的段落排在前面，却把「退货政策」挤到了边缘，还混进了无关的「配送时效」。它只顾住了一半。
- **关键词检索**靠「生鲜」「退货」这些精确词，兜住了退货那一半。
- **只有混合检索**同时抓住了运费**和**退货——两个都命中（✅ 全部命中）。

这就是混合的全部价值：**一路会漏的，另一路补上。** 单路检索总有它照不到的角落，混合把两路的视野叠起来，漏得最少。

## 2026 的生产共识

到 2026 年，业界对 RAG 检索这一步的共识已经相当统一：**纯语义不够，纯关键词也不够，「混合检索 + 重排」是标配。** 混合检索负责「别漏」（尽量把相关的都捞回来），重排负责「排对」（把最相关的顶到最前）——重排就是下一章 [K4](../04-reranking/index.md) 的主题。

:::caution 混合检索也不是银弹
两点提醒。其一，**混合不是无脑叠加**：两路的相对权重、RRF 的常数、各召回多少条，都是要拿真实查询调的参数（回扣 [K2](../02-chunking/index.md) 的「评测驱动调参」，也预告 [K6](../06-evaluation/index.md)）。其二，也是最根本的——**混合检索救不了「知识库里根本没有答案」**。它只能在**已有**的资料里尽量捞全；库里缺了、写错了，两路一起上也变不出正确答案（回扣 [K0](../00-why-rag/01-three-flaws.mdx)：查不到照样答不好）。检索技术再花哨，也盖不过知识库本身的质量。
:::

<DeepDive title="RRF 公式与其他融合策略">

**RRF 的公式。** 一个文档 $d$ 的融合分，是它在每一路排名里「名次倒数」的和：

$$
\text{RRF}(d) = \sum_{i} \frac{1}{k + \text{rank}_i(d)}
$$

其中 $\text{rank}_i(d)$ 是文档 $d$ 在第 $i$ 路检索里的排名（第 1 名 rank=1），$k$ 是一个常数（常取 60）。逐项理解：

- **名次越靠前，倒数越大**：排第 1 贡献 $\frac{1}{k+1}$，排第 10 贡献 $\frac{1}{k+10}$，往后飞快衰减——所以头部名次最值钱。
- **$k$ 起什么作用**：它压平了头部的差距。若没有 $k$（即 $k=0$），第 1 名 vs 第 2 名会是 $1$ vs $0.5$，差一倍太悬殊；加上 $k=60$，变成 $\frac{1}{61}$ vs $\frac{1}{62}$，差距温和，让「进了前几名」这件事本身比「具体第几」更重要，更鲁棒。
- **在两路都靠前的文档赢**：一个文档若在关键词和语义两个榜里都排前列，两个较大的倒数相加，总分自然高——这正是我们想要的「两路都认可」。

**走一个完整的小例子（$k=60$）。** 三段文档 A、B、C，两路各出一个榜：

| 文档 | 关键词榜名次 | 语义榜名次 | RRF 分（$\frac{1}{60+r_{词}}+\frac{1}{60+r_{语}}$） |
| --- | --- | --- | --- |
| A | 第 1 | 第 3 | $\frac{1}{61}+\frac{1}{63} = 0.0164+0.0159 = 0.0323$ |
| B | 第 2 | 第 1 | $\frac{1}{62}+\frac{1}{61} = 0.0161+0.0164 = 0.0325$ |
| C | 第 3 | 第 2 | $\frac{1}{63}+\frac{1}{62} = 0.0159+0.0161 = 0.0320$ |

融合后最终排名：**B（0.0325）> A（0.0323）> C（0.0320）**。注意 B 赢了——它不是任何单路的第 1（关键词榜第 1 是 A），但它在两路都稳居前二，「两路都认可」让它综合胜出。这正是 RRF 想要的效果。

**为什么 RRF 比「加权求和分数」更鲁棒。** 直接把两路的原始分加权相加，要先归一化，而归一化对异常分数（某条 BM25 分特别高）很敏感，容易被一路带偏。RRF 只用名次，天然免疫这些问题——这是它成为工业默认的原因。代价是丢掉了「分数间隔」信息（第 1 名比第 2 名强多少），但实践证明这点损失换来的鲁棒性很值。

**加权 RRF 与其他融合。** 可以给某一路更高权重（比如你的场景更吃精确匹配，就给关键词加权）。除 RRF 外还有 CombSUM/CombMNZ（基于归一化分数）等融合法，但 RRF 因简单鲁棒最常用。

**工程实现。** 有的向量库（如支持混合检索的那些）原生提供混合检索 + RRF，一次调用出结果；有的则要你自己跑两路、再在应用层做 RRF。同时维护「倒排索引（关键词）+ ANN 索引（向量）」两套索引，是混合检索的基础设施成本。

**目标是召回，不是精度。** 想清楚混合检索在流水线里的定位：它主要为了**提高召回率**——尽量别漏掉相关文档。因为漏掉的东西，后面 [K4 的重排](../04-reranking/index.md)也救不回来（重排只能在召回的候选里重新排序）。所以混合这一步「宁滥勿缺」，把该捞的都捞回来，精挑细选交给下一棒。

**RRF 融合，其实就几行。** 上面那张手算表，翻译成代码是这样——`k=60`，两路各出一个「文档 id 按名次排好」的列表，按名次倒数累加即可：

```python
def rrf_fuse(*rankings, k=60):
    """rankings: 若干路检索的结果，每路是按相关度排好的文档 id 列表。"""
    scores = {}
    for ranking in rankings:                 # 关键词榜、语义榜……
        for rank, doc_id in enumerate(ranking):  # rank 从 0 开始
            scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)  # 融合分从高到低

# 关键词榜 [A, B, C]、语义榜 [B, A, D] → 融合后 B 综合最高
fused = rrf_fuse(["A", "B", "C"], ["B", "A", "D"])
```

</DeepDive>

## 小结

:::tip 本节要点
- **混合检索** = 关键词 + 语义两路一起跑，各补盲区，再融合——2026 年生产 RAG 的标配。
- 融合的难题：两路分数刻度不同、没法直接加。**RRF** 的解法是**只看名次不看分数**，把不可比的分数翻译成通用的排名。
- 混合的价值在「少漏」：跨话题、单路会漏一半的查询，两路合力才能都命中。
- 它救不了「库里没答案」，两路权重也要调——检索技术盖不过知识库本身的质量。
:::

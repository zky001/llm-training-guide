# K1.1 从「对字」到「对意思」

## 老式搜索的两个死穴

想想你平时怎么用搜索框：输入「退货」，系统去找所有含「退货」二字的文档。这叫**关键词检索**（经典算法是 BM25，几十年来搜索引擎的主力）。它简单、快、可解释——但拿来做 RAG，有两个致命死穴。

**死穴一：换个说法就找不到。** 用户不会照着你文档里的用词提问。你的手册写「七天无理由退换」，用户却说「买的东西不想要了怎么办」——两句话意思完全一样，却**一个共同的字都没有**。关键词检索对着字面死磕，直接漏掉。这就是「同义不同字」问题，中文里尤其严重（一个意思有无数种说法）。

**死穴二：一词多义分不清。** 用户搜「苹果」，他到底想要**水果**，还是**手机配件**？关键词检索只认「苹果」这两个字，会把两类文档一股脑全捞上来，分不清语境。

这两个死穴，根子是同一个：关键词检索只会**对字**，不会**对意思**。

## 核心思想：把意思变成坐标

RAG 需要的是<Term id="semantic-search">语义检索</Term>——按「意思」找，而不是按「字」找。怎么做到？答案你在上篇其实已经见过了。

上篇 [1.4 节](/docs/language-models/embedding)讲过一个神奇的事实：把词变成向量之后，**意思相近的词，坐标也相近**（「猫」和「狗」离得近，和「汽车」离得远）。语义检索把这个思想放大到整段话：

> 用同一个模型，把**用户的查询**和**知识库里的每段资料**都变成向量（一串坐标）。于是「找意思最相近的资料」，就变成了一道几何题——**「找坐标离查询最近的那个点」**。

「买的东西不想要了」和「七天无理由退换」，字面毫无重合，但它们的**意思**几乎一样，所以变成向量后**坐标也几乎重合**——语义检索一下就能对上。

请务必做两个对比实验：

**实验一：同义不同字。** 默认查询就是「买的东西不想要了」。先切到「🔤 关键词检索」——你会看到它**一个词都没命中**「退货政策」那段（字面零重合），只能瞎排。再切回「🧭 语义检索」——它精准命中了。这一正一反，就是语义检索存在的全部理由。

**实验二：一词多义。** 选查询「我想买苹果」。关键词检索会把**冰糖心苹果**（水果）和**苹果手机充电线**（配件）都捞上来——都含「苹果」二字，它分不清。切回语义检索，它能根据「想买」的语境，把水果排到前面。

面板上那个「相似度」百分比，衡量的就是查询和资料「意思有多近」。它背后的正式指标叫<Term id="cosine-similarity">余弦相似度</Term>——衡量两个向量方向有多接近的标尺，越接近 1 越像（数学细节见深入层）。

:::caution 语义检索是万能的吗？
不是。它有一个和关键词检索正好相反的短板：**遇到「必须精确匹配」的东西会犯迷糊**。用户查产品型号「X9-2000」、订单号、身份证号、某个专有名词——这些要的是**一字不差**，而语义检索追求「意思相近」，反而可能把「X9-2000」和「X9-3000」当成差不多的东西混起来。这种场景，老实的关键词检索反而更准。所以现实中最好的做法往往不是二选一，而是**两个一起用**——这就是 [K3 混合检索](../03-retrieval-methods/index.md)要讲的。另外提醒：实验里的二维坐标是手工构造的概念示意，真实向量有几百上千维，没法直接画出来。
:::

<DeepDive title="嵌入模型与余弦相似度">

**谁把文本变成向量？** 一个专门的**嵌入模型**（embedding model）。它和上篇 [1.4 节](/docs/language-models/embedding)的词向量同源，但更进一步——它把**整段话**编码成一个向量（不只是单个词），且经过专门训练，让「语义相近的句子」向量也相近。关键是：**查询和文档必须用同一个嵌入模型编码**，否则两者的坐标系对不上，比距离就没意义了。

**余弦相似度的公式。** 衡量两个向量 $a$、$b$ 有多「同向」：

$$
\cos(a, b) = \frac{a \cdot b}{\lVert a \rVert \, \lVert b \rVert}
$$

分子是点积，分母是两个向量的长度。结果落在 $[-1, 1]$：越接近 1 越同向（意思越像），接近 0 表示无关。**为什么用余弦而不用直线距离？** 因为它**只看方向、不看长度**——一段话说三遍，向量会变长，但意思没变，方向也不变。余弦相似度天然忽略这种「长度噪声」，只抓「语义方向」。

**代入数字走一遍。** 拿两个二维向量手算一次就懂了：$a = [1, 0]$、$b = [0.6, 0.8]$。

- 点积 $a \cdot b = 1 \times 0.6 + 0 \times 0.8 = 0.6$；
- 长度 $\lVert a \rVert = \sqrt{1^2 + 0^2} = 1$，$\lVert b \rVert = \sqrt{0.6^2 + 0.8^2} = \sqrt{1} = 1$；
- 于是 $\cos(a, b) = 0.6 / (1 \times 1) = 0.6$。

0.6 表示两者方向有点接近、但不算很像。真实系统里向量是几百上千维，道理完全一样——只是维度多、没法手画而已。

**双编码器：为什么能扛亿级检索。** 语义检索的标准架构叫**双编码器**（bi-encoder / dual-encoder）：查询和文档**各自独立**编码成向量。这带来一个巨大的工程红利——文档向量可以**离线预先算好、存进库里**；检索时只需实时编码「查询」这一个短句，再去库里比距离。若查询和文档必须成对一起过模型（那叫交叉编码器），亿级文档每次查询都要重算，根本跑不动。双编码器把「编码」和「比对」解耦，才让大规模语义检索成为可能（[K4](../04-reranking/index.md) 会看到，交叉编码器精度更高但只能用在小范围重排）。

**嵌入模型本身要选型。** 嵌入模型不是唯一的——2024–2026 年有一批专门为检索训练的嵌入模型，中英文各有强者，效果差异明显。选错嵌入模型，后面切块、检索做得再好也白搭，所以它是 RAG 选型的第一个关键决策。

**一个实用简化：** 如果把所有向量都归一化到单位长度（长度 = 1），余弦相似度就退化成了简单的**点积**——这也是为什么很多向量库内部直接用点积算相似度、又快又等价。

**几行代码算余弦相似度**（`emb` 是你的嵌入模型）：

```python
import numpy as np

def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

q = emb("买的东西不想要了")          # 查询向量
d = emb("支持七天无理由退换")        # 文档向量
print(cosine(q, d))                 # 越接近 1，意思越像

# 若向量已归一化（norm=1），直接点积即可，等价又更快：
# print(np.dot(q, d))
```

## 小结

:::tip 本节要点
- 关键词检索（BM25）只会「对字」，两个死穴：**同义不同字找不到**、**一词多义分不清**。
- 语义检索把查询和资料都变成向量，「找意思最近的」= 「找坐标最近的点」——回扣上篇「意思相近→向量相近」。
- 衡量「意思有多近」用**余弦相似度**：只看方向不看长度，越接近 1 越像。
- 语义检索也有短板：**精确匹配**（型号、订单号）反而是关键词的主场——所以现实常两者并用（K3）。
:::

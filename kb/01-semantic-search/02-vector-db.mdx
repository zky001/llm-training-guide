# K1.2 向量数据库：在亿万个坐标里秒级找最近邻

## 「找最近的点」有个隐藏的坑

[上一节](./01-from-keyword-to-semantic.mdx)说，语义检索的本质是「找坐标离查询最近的点」。在实验里那个只有 11 段资料的小知识库，这没问题——挨个算一遍距离，11 次比较，一眨眼就完。

但真实的知识库不是 11 段。一家公司的全部文档、一个网站的所有页面，切完块动辄**几百万上千万段**。如果每来一个用户查询，都要和库里**全部**向量各算一次距离，那就是几百万次计算——慢到用户等得睡着。这个「和全部逐一比对」的笨办法叫**线性扫描**，库越大越慢，根本没法用在生产上。

于是需要一个专门的工具来解决它。

## 向量数据库登场

<Term id="vector-db">向量数据库</Term>就是专门为这件事造的：它存海量向量，并能**极快地**找出「离给定查询最近的几个邻居」。它不像普通数据库那样按「等于/大于」查行，而是按「谁离我最近」查向量。

它靠什么快？靠一个关键的妥协——**不追求「最近」，只追求「几乎最近」**。

## 近似最近邻：用一点点精度换巨大的速度

这个妥协有个正式名字：<Term id="ann">近似最近邻（ANN，Approximate Nearest Neighbor）</Term>。

它的思路是：精确地找到「那个绝对最近的点」代价太高；但如果允许**偶尔漏掉真正第一名、返回一个「几乎最近」的**，速度就能快上几个数量级。对 RAG 来说这笔买卖太划算了——检索本来就要返回好几段（top-k），个别名次错一位、极偶尔漏掉一个边缘结果，对最终答案几乎没影响，换来的却是从「几百万次比对」到「几十次比对」的飞跃。

这和上篇 [6.3 节量化](/docs/inference/quantization)是同一种工程哲学：**用一点点精度，换巨大的效率**。整个 AI 工程里，这种「够用就好」的取舍无处不在。

打个比方帮你记住 ANN 的聪明之处：

> 你要找「离你最近的餐馆」。笨办法是量出你到全城每一家餐馆的距离，再挑最小的——这是线性扫描。聪明办法是：城市早就按**区、商圈**分好了层，你只需要「先定位到自己所在的商圈，再在这一小片里找」。ANN 就是给向量空间提前建好了这样一张**分区地图**，让你不必走遍全城。

## 向量库不只是「快」

除了 ANN 索引，一个生产级向量数据库通常还负责：

- **存向量 + 元数据**：每段向量旁边挂着「来自哪篇文档、哪一节、什么日期」等信息（这些元数据在 [K5 引用](../05-generation/index.md)和下面的过滤里都要用）；
- **带过滤的检索**：「只在 2025 年之后的文档里找」「只在『财务』分类里找」——先按元数据缩小范围，再在里面找最近邻；
- **增删改**：文档更新了，能高效地插入新向量、删除旧的（回扣 [K0](../00-why-rag/01-three-flaws.mdx)：知识库随时可更新，正是 RAG 的核心优势）。

主流选型（2025–2026 年生态，不展开）：专用向量库如 Milvus、Qdrant、Weaviate；也有传统数据库/搜索引擎加向量扩展的路线，如 PostgreSQL 的 pgvector、Elasticsearch——后者的好处是「一套系统同时做关键词和向量检索」，为 [K3 混合检索](../03-retrieval-methods/index.md)提供便利。

:::caution 向量库快 ≠ 结果准
两个必须澄清的点。其一：ANN 是**近似**——它极少数情况下会漏掉真正最近的那个。生产中要在「召回率」（找全的比例）和「速度」之间调参数，追求极致召回就得牺牲速度，反之亦然，没有免费的午餐。其二，也更重要：**向量库只解决「快速找到相似向量」这一件事**。至于找回来的东西对不对、全不全，取决于前面两步——嵌入模型好不好（[K1.1](./01-from-keyword-to-semantic.mdx)）、切块切得对不对（[K2](../02-chunking/index.md)）。库再快，喂进去的是烂向量，捞出来的也是烂结果。**别把「检索快」误当成「检索好」。**
:::

<DeepDive title="ANN 索引的两大家族">

**为什么线性扫描不可行。** 每次查询要和全部 $N$ 个向量比对，复杂度 $O(N)$。$N$ 上百万、每个向量上千维时，单次查询就是十亿级浮点运算——延迟高到不可接受，而且并发一上来直接压垮。ANN 索引把单次查询的比对次数从 $O(N)$ 降到近似 $O(\log N)$ 量级。

**家族一：HNSW（分层可导航小世界图）。** 当前最主流。直觉：给向量建一张**分层的图**，像一个「高速公路 + 小路」的多层网络——顶层是稀疏的高速公路（少数节点、长距离跳跃），底层是密集的小路（覆盖所有点、短距离精调）。查询时从顶层高速公路快速「跳」到目标大致区域，再逐层下沉到小路精细定位。查得准、快，代价是内存占用较大（要存图结构）。

**家族二：IVF（倒排文件）。** 先用聚类把所有向量分成若干「桶」（就是前面比喻里的「商圈」）；查询时先判断落在哪几个桶，只在这几个桶里精找，跳过其余。它常和 **PQ（乘积量化）** 搭配——把每个向量压缩成更短的编码来省内存和加速，这正是上篇 [6.3 量化](/docs/inference/quantization)的思想用在了检索上。IVF+PQ 内存友好，适合超大规模，代价是精度略低、需要调桶数等参数。

**三角权衡。** ANN 索引本质是在**召回率、延迟、内存**三者间取舍：想要高召回就得多搜几个桶/多探几个邻居（更慢），想省内存就得压缩向量（更不准）。没有一个索引在三个维度上全赢，选型取决于你的规模和场景。

**混合过滤的工程难点。** 「向量检索 + 元数据过滤」看着简单，实现却棘手：是「先过滤再在子集里建临时索引找最近邻」（过滤强但可能慢），还是「先找最近邻再扔掉不满足过滤的」（快但可能召回不足）？各家向量库的策略不同，是评估选型时值得深挖的一点。

**长这样：一次带过滤的检索。** 入库时每段向量都挂上元数据（一个小 JSON），检索时就能先按元数据缩小范围、再找最近邻：

```python
# 入库：向量 + 元数据一起存
collection.insert(
    vector=emb(chunk_text),
    metadata={"source": "员工手册v3.2", "date": "2025-06", "section": "6", "lang": "zh"},
)

# 检索：只在「2025 年之后的中文文档」里找最近的 5 段
hits = collection.search(
    query_vector=emb("退货运费谁承担"),
    top_k=5,
    filter={"date": {">=": "2025-01"}, "lang": "zh"},
)
```

这个 `filter` 就是「只查最新的、只查某分类」的落地方式——也是知识库能「按时效、按权限」筛选的基础。

</DeepDive>

## 小结

:::tip 本节要点
- 语义检索要「找最近的点」，但库有几百万段时，和全部逐一比对（线性扫描）慢到没法用。
- **向量数据库**专门解决这个：存海量向量并极快找最近邻，靠的是**近似最近邻（ANN）**。
- ANN 的核心妥协：**用一点点精度换巨大的速度**（和上篇量化同源）——「几乎最近」对 RAG 足够。
- 向量库快 ≠ 结果准：它只负责「快速找相似向量」，找得对不对仍取决于嵌入模型和切块质量。
:::

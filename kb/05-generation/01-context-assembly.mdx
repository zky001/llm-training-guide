# K5.1 上下文拼装：把资料摆到模型面前

## 摆盘也是门手艺

检索出来的几段资料，不能就那么原封不动地甩给模型。**怎么摆**——用什么顺序、什么格式、配上什么指令——直接决定模型看不看得懂、听不听话。

这一步叫 <Term id="context-augmentation">上下文拼装</Term>，是 RAG 三步里「增强」这个字的落地（回扣 [K0.2](../00-why-rag/02-what-is-rag.mdx)）。它做的事说来简单，却步步是坑：

1. **把检索到的块编号列好**——`[1] …… [2] ……`，方便模型引用、也方便你核对；
2. **配上明确的指令**——「只根据以下资料回答」「资料里没有的，就说没有」；
3. **把用户的问题放在末尾**——让模型读完资料，带着问题作答。

拼好的这一大段文本（指令 + 资料 + 问题），就是真正送进模型的提示词。

几个值得注意的地方：

- **第 ③ 步先清了噪声**：检索命中的三条里，「支付方式」明显跑题，被重排淘汰了（回扣 [K4](../04-reranking/index.md)）。拼装前先把噪声清掉，模型才不会被带偏。
- **第 ④ 步的两句指令**：「请仅根据以上资料回答」和「在每句话后标注来源编号」——正是抑制幻觉、逼出引用的关键。少了它们，模型就容易放飞自我（下一节细讲）。
- **资料是编号的**：`[1] 会员权益…… [2] 积分规则……`，这样模型答的时候才能标「〔1〕〔2〕」，你也才能一眼对回原文。

## 拼装的三个坑

摆盘摆不好，好料也白搭。三个最常见的坑：

**坑一：放太多块。** 「多多益善」是错觉。检索到的块不是塞得越多越好——塞满上下文会稀释重点、拖慢速度、还烧钱（回扣上篇 [8.2 长上下文](/docs/frontier/long-context)的成本，和 [K2](../02-chunking/index.md) 切太大的道理一样）。宁可少而精，让每一块都真的相关。

**坑二：顺序有讲究。** 模型对上下文的**首尾更敏感**，塞在正中间的信息反而容易被忽略——这个现象叫「lost in the middle」（迷失在中间，2023 年的研究）。所以经验做法是：**把最相关的块放在最前面和最后面**，别把关键证据埋在中段。

**坑三：指令不明确。** 如果你只把资料一摆、问题一抛，不加约束，模型就会默认「自由发挥」，掺进它训练时的旧记忆。一句明确的「只根据资料回答，没有就说没有」，往往比换个更大的模型还管用。

:::caution 这一步没有唯一正确答案
拼装的格式、顺序、指令措辞，都要按你用的模型和任务去调——没有一套模板包打天下。更要命的是：**「把资料摆好」不等于「模型会用好」**。就算你把对的资料端端正正摆在它面前、指令也写清楚了，模型有时仍会走神、仍会掺记忆。这是下一节 [K5.2](./02-grounding-citation.mdx) 要专门对付的问题——拼装是「把料摆好」，有据回答是「逼它照料答」。
:::

<DeepDive title="提示词模板与上下文预算">

**一个典型的 RAG 提示词模板**长这样：

```text
[System] 你是一个严谨的问答助手。只根据「参考资料」回答；
资料未提及的信息，回答「资料未提及」，不要用你自己的知识补充。
回答时在每个论断后标注来源编号，如〔1〕。

[参考资料]
[1] {来源标题} · {日期}：{chunk 正文}
[2] {来源标题} · {日期}：{chunk 正文}
...

[问题] {用户的问题}
```

三段式：**系统指令**（定规矩）+ **编号资料**（带来源标题和日期，为引用和时效判断铺路）+ **问题**（放最后）。

**上下文预算分配。** 上下文窗口是有限且花钱的（回扣上篇 [6.2 KV Cache](/docs/inference/kv-cache) 的显存账和 [8.2 长上下文](/docs/frontier/long-context) 的成本）。你得给检索块、对话历史、模型的生成各留一份预算——检索块塞太多，留给生成的就少了，还可能触发「lost in the middle」。常见做法是限定「最多放 N 个块」或「资料总长不超过窗口的某个比例」。

**块的位置与 lost-in-the-middle。** 2023 年的研究发现，模型对上下文首尾的信息利用率明显高于中段。所以重排（[K4](../04-reranking/index.md)）之后，工程上常把最相关的块放在最前和最后，把次要的放中间——用位置对冲「中间盲区」。

**块之间的分隔与元数据。** 每个块前面标清楚 `[编号] 来源标题 · 日期`，既让模型能准确引用，也让它能判断时效（「这条是 2020 年的，那条是 2026 年的」）。分隔清晰还能防止模型把两个块的内容张冠李戴。

**多轮对话里的共处。** 真实的问答往往是多轮的。这时上下文里既有对话历史、又有本轮检索的资料——两者要分区摆放、并让模型分清「哪些是历史对话、哪些是这次查到的资料」，否则容易混淆。

**拼装一个提示词，就是一段 f-string。** 把检索到的块编号列好、配上「只根据资料答」的指令、问题放末尾：

```python
def build_prompt(question, chunks):
    context = "\n".join(f"[{i+1}] {c}" for i, c in enumerate(chunks))
    return (
        f"参考资料：\n{context}\n\n"
        f"请仅根据以上资料回答，并在每句话后标注来源编号 [n]；"
        f"资料未提及的，回答「资料未提及」。\n\n"
        f"问题：{question}"
    )
```

编号 `[1] [2]` 是给下一节「引用」用的锚点，那两句指令则是抑制幻觉的关键——别小看这几行 f-string，RAG 答得靠不靠谱，一大半就压在这里。

</DeepDive>

## 小结

:::tip 本节要点
- **上下文拼装**是「增强」的落地：把检索块编号列好、配上明确指令、问题放末尾，拼成送进模型的提示词。
- 三个坑：**放太多块**（稀释重点、烧钱）、**顺序不讲究**（lost in the middle，最相关的要放首尾）、**指令不明确**（模型放飞自我掺记忆）。
- 那两句指令「只根据资料答」「标注来源」，往往比换个更大的模型还管用。
- 「把资料摆好」不等于「模型会用好」——这要靠下一节的有据回答来兜底。
:::
